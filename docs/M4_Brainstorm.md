# PicZTream (PZT) Milestone 4 头脑风暴（Agent 半自动 · 即时通讯接入）

## 文档定位

这是 M4 的**头脑风暴文档，不是 PRD、更不是 Eng Design**。目的是把"agent 能做什么、技术栈长什么样、本地模型怎么用、有哪些必须先拍板的岔路"探索清楚并固化下来，作为后续正式 PRD 的输入。里面很多内容是可能性列举和方向探讨，带明确"待拍板/待确认"标记的条目还没有结论，不要当成已经定下来的设计去实现。

写作时间：2026-07-13。此时 M0-M2 已完成，M3（选片辅助评分 + 近似重复检测两个增量）已完成并经过真机验证与一轮 E2E 反馈修复。

## 一、M4 在 Roadmap 里的定位（先对齐前提）

`docs/Roadmap.md` 对 M4 的定义：**复用前几阶段沉淀的 `core` 库能力，新增一个 headless 编排层**，接收即时通讯工具发来的照片，自动完成初筛、打标签、生成候选文案，结果推送回用户做最终确认，**人工始终保留最后拍板权**。

三个关键前提，后面所有设计都不能违背：

1. **架构上，agent 层是 `cli/` 的平级兄弟**，两者都只调 `core/api`，互不依赖。`core/` 当初就明确要求"保持对未来非 cli 调用方（如 headless agent 层）的可复用性"——M4 正是那个被预留的调用方。IM 传输、网络、会话状态这些**绝不能渗进 `core/`**。
2. **M3 与 M4 无先后依赖**。Roadmap 原文："即便 Milestone 3 尚未完成，也可以先用简单规则跑通编排链路，两者互为增强而非互为阻塞。"这给了 M4 一条低风险起步路径：先用最笨的规则串起编排链，再逐步把 M3 的 AI 能力和本地模型接进去。
3. **人工保留最终拍板权**。agent 永远是"提议者"，不是"执行者"。整条流程以"把结果发回给你确认"收尾，绝不自动发布到任何平台。

### 与 AI 使用边界的关系（必须澄清）

AGENTS.md 的"AI 使用边界"写着：LLM 不参与任何需要保证确定性和实时性能的核心执行路径，不得在性能关键代码中依赖运行时调用 LLM 决策。

M4 大量用运行时 LLM，会不会违背这条？**不违背**，前提是守住三条线：

- 零延迟保证只约束 `pzt open` 的交互式选片主循环。**agent 编排层根本不是那条路径**——它是批处理、异步、末端人工确认的独立执行上下文，慢几秒几十秒都无所谓。
- `core/` 继续保持纯确定性、无 LLM。agent 层调 LLM 拿到的结果，要么转成 `core` 认识的结构化输入（recipe 参数、标签、选中的 image_id 列表），要么就是给人看的文案——`core` 自己永远不知道有 LLM 存在。
- LLM 输出始终是"建议"，经人工确认才生效，跟 M3 的评分/去重结果"写回数据库供参考、不替代用户判断"是同一个精神。

## 二、主用例逐步拆解（WhatsApp → 朋友圈九宫格）

用户构思的核心用例：一天旅行拍了几十张，全发给某个 WhatsApp 账号，交互中说明"要发朋友圈、想要 XX 风格"，agent 自动建 project → AI 送评 → dedup → 从"通过评估 + 非废片 + 非重复"里挑 9 张 → 打标签 → 统一风格 → 写文案 → 把图片和文案发回。

逐步拆解，每步标注**复用了哪些现成 `core` 能力 / 什么是全新的 / 注意点**：

| 步骤 | 复用 core | 全新能力 | 注意点 |
|---|---|---|---|
| 1. 从 WhatsApp 接收图片 + 意图 | 无 | IM 传输层 + 意图 NLU | WhatsApp 媒体是重压缩 JPEG、常被剥 EXIF；官方 vs 非官方 API 有 ToS/封号风险（见第六节）；"发朋友圈/想要暖色胶片风/要 9 张"这段自然语言要解析成结构化意图 |
| 2. 自动建 project | `create_project(name, folder, ...)` | agent 把下载的媒体落到暂存目录再调它 | 项目命名（按日期？按用户消息？）；暂存目录管理；WhatsApp 图基本都是 JPEG，不涉及 RAW 分支 |
| 3. AI 送评（曝光/构图/对焦打分） | `EvaluationWorker` / `request_evaluation` / `evaluated_image_ids` | 无（直接复用 M3） | **token 成本爆点在这里**：几十张 × 每张一次多模态调用。这是本地模型的首要动机（第五节）。WhatsApp 压过的图作为质量评估输入本身已经劣化 |
| 4. 去重 | `find_and_tag_duplicates` | 无（直接复用 M3） | 本地算法（dHash），无 API 成本。但 **EXIF 被剥 → captured_at 缺失 → 时间聚类退化**（正好命中 F-08 的 skipped_no_capture_time） |
| 5. 从（通过评估 + 非废片 + 非重复）挑 9 张 | `passes_gate` / `overall_score` / `images_with_tag` / `get_image` | **多样性感知的策展算法**（见第七节） | 朴素 top-9 by score 会挑出 9 张几乎一样的照片。要"讲好一天的故事"需要覆盖度/多样性，不只是分数排序 |
| 6. 打标签 | `create_tag` / `add_tag` / `ensure_reject_tag` | 无 | 给选中的 9 张打个"发布/精选"标签，纯复用 |
| 7. 统一风格 | `set_image_recipe` / `render` / `list_presets` / `create_version` / `export_images` | **自然语言风格 → recipe 参数**的生成 | recipe 只有色调/白平衡（highlights/shadows/wb_shift）+ 3D LUT，**没有几何变换（裁切/旋转）**——如果"风格"含构图矫正，会重新激活 M3 的几何变换缺口（第八节）。生成结构化 recipe 是 AGENTS.md 明确允许的 LLM 用法 |
| 8. 写文案 | 无 | **多模态看图 + 中文社媒文案生成** | 需要强中文生成能力和社媒语气；量小（一批一次），云端成本可接受；用户要能改（人工拍板） |
| 9. 发回图片 + 文案 | `export_images`（导出成品 JPEG） | IM 回传 + 人工确认回环 | agent 交付素材，**不能替用户发朋友圈**（微信 Moments 无 API）；"换掉第 3 张，太暗了"这类对话式微调 |

**核心洞察**：这条流水线的绝大部分（建项目、送评、去重、标签、风格、导出）M3/M1/M0 已经全建好了。M4 真正**新增的表面很小**，集中在边缘：(a) IM 传输，(b) 意图 NLU，(c) 多样性策展"挑 9 张"，(d) 风格→recipe 生成，(e) 文案生成，(f) 编排状态机本身。agent 主要是**把现成的 `core/api` 操作按顺序串起来**，加上少数几个新的边缘能力。

## 三、其它可能的用例（模拟用户探索）

以"重度拍照 + 有选片/交付需求的人"这个真实用户视角，横向想一批用例，标注各自压测系统的哪块、以及 v1 该不该做：

| 用例 | 一句话 | 压测哪块 | v1? |
|---|---|---|---|
| 朋友圈九宫格（主用例） | 策展 + 风格 + 文案，端到端 | 全链路 | 目标形态 |
| 纯粗筛交付 | 发 200 张活动照，只要留下的（去重 + 去废 + top 挑选），不要风格/文案 | 送评 + 去重 + 策展 | **是**（最省、最快，验证编排最小闭环） |
| 纯批量风格化 | 我已经选好了，只帮我统一套个风格发回 | recipe 生成 + 应用 + 导出 | 较易 |
| 摄影师给客户初筛 | 发一次拍摄，agent 产出初筛 gallery 给客户过 | 送评 + 策展 + 交付 | 贴合 PZT 的准专业定位（RAW/recipe/有序标签都是为这类人建的） |
| 检索式提问 | "这些里哪张孩子拍得最清楚" | 语义理解 + 检索 | later（需要 embedding/主体理解） |
| 增量续拍 | 一天里陆续丢照片，最后说"收工"再出结果 | 会话状态持久化 | later（状态机更复杂） |
| 多目的地 | 同一批：9 张发朋友圈、20 张进相册、3 张发 IG | 目的地感知的差异化策展 | later |
| 批量整理 | "把最近 500 张按事件分组打标签"，无社交输出 | 语义聚类 + 自动打标签 | later（正好接住 M3 未做的"自动打标签"） |
| 对话式微调 | "换掉第 3 张""文案再活泼点""风格更冷一些" | 人工确认回环 | v1 就要有基础版（否则违背"人工拍板"） |

结论：**v1 应该锁定"纯粗筛交付"这个最小闭环**（送评 → 去重 → 策展 → 导出交付），把风格化、文案、丰富对话作为增量往上叠。这跟 Roadmap"先用简单规则跑通编排链路"完全一致。

## 四、贯穿性技术决策与注意点

### 4.1 编排层用什么语言 / 怎么调 core（**最大的一个待拍板项**）

这是 M4 的头号岔路，直接决定其它一切：

- **选项 A：agent 层也用 C++**，直接链接 `core`。优点：单一工具链、类型直通、零 IPC 开销。缺点：WhatsApp 生态（whatsapp-web.js/Baileys）和本地模型生态（MLX、Ollama Python 客户端、Apple Vision 的 Swift/pyobjc 绑定）几乎都是 Python/JS/Swift 原生，C++ 里要么自己封 HTTP，要么起桥接进程。
- **选项 B：agent 层用 Python/TS/Swift**，通过一层薄边界调 `core`（把 `core` 能力经 CLI 子命令 / 本地 IPC / FFI 暴露）。优点：IM 和本地模型生态原生可用、迭代快。缺点：多一套工具链、多一层进程/序列化边界、`core/api` 可能要补一批 headless 友好的入口（现在很多能力只经 `cli` 触发）。

我的倾向：**编排层和外部集成（IM、模型服务）用 Python/Swift，`core` 通过明确的进程/IPC 边界暴露一组 headless 命令**——理由是 M4 的价值在"编排 + 集成"，而集成生态在 C++ 之外；硬用 C++ 会把大量精力耗在给 WhatsApp/MLX 封边界上，违背"复用 core、别重造轮子"的初衷。但这是**用户必须拍板的决定**，牵涉工具链偏好和长期维护成本。

### 4.2 WhatsApp 接入 + "控制通道 vs 像素通道"分离

- 官方 WhatsApp Business Cloud API：合规，但有审批门槛和计费。非官方库（whatsapp-web.js/Baileys）：好上手，但违反 ToS、有封号风险。**待拍板**。
- **一个重要洞察**：WhatsApp 的图是重压缩 JPEG、常剥 EXIF——作为"要认真选片和调色"的原始素材质量不够。可以把两件事拆开：**WhatsApp 只做控制通道**（对话、意图、发回预览/文案），**全分辨率原图走另一条像素通道**（Mac 本机照片库 / 一个被监听的同步文件夹 / AirDrop）。控制与像素解耦，既绕开 WhatsApp 画质问题，也让 v1 可以先用"监听文件夹 + CLI 触发"跑通、把 IM 接入延后。

### 4.3 其它贯穿注意点

- **确定性 / 人工回环**：agent 只提议不发布，流程以确认收尾。
- **可恢复性**：批处理耗时长（几十张送评可能数分钟），进程崩溃或用户离线后编排状态要能续跑。DB 已经持久化了评估/标签（状态天然耐久），但**编排状态**（走到哪步、用户要什么）需要自己持久化。这正好接住 M3 未做的"会话续点"思路（F-24）。
- **隐私 / 数据出机**：照片上云是 M3 就点出的顾虑。本地优先分层（第五节）能大幅缓解，但文案/策展可能仍要把缩略图发上云——应该是**有意识、可披露、可最小化**的选择（只发降采样缩略图、只发最终 9 张而非全部原图）。
- **进度反馈**：即便本地，2B VLM 跑几十张也要时间，得通过 IM 通道回报进度（"已评估 12/40..."），复用 `ScanProgressFn`/`EvaluationWorker` 的进度模式。
- **成本上界思维**：真正贵的是**逐张**的评估（几十次多模态调用/批），文案只是 1 次。省钱的关键不是"全部本地化"，而是"把高频的逐张任务本地化、把一次性的创作任务留云端"（第五节的核心原则）。

## 五、本地模型设计（重点 · 教学向）

用户明确说 remote API 的 token 量扛不住、且对本地模型不熟。这一节先讲清楚"在 Mac 上跑本地模型到底是怎么回事"，再给出分层设计。

### 5.1 先破一个误区：不是"把一个大模型搬到本地"

正确的思路是**按任务分层**，让每个 AI 子任务落到"刚好够用、又最省"的那一层。M4 里的 AI 任务需求差异巨大：

- 逐张打分：**高频**（每张都要），但判断相对客观。
- 语义标签（沙滩/美食/人像）：中频，分类任务。
- 挑 9 张：一批一次，是"分数 + 相似度"上的算法，未必要 LLM。
- 风格→recipe：一批一次，小结构化输出。
- 写文案：**一批一次**，但要强中文创作力。

把它们塞进同一个"大模型"既贵又浪费。分三层：

### 5.2 三层策略

**Tier 0 — 经典 CV / Apple 原生框架（端上、确定性、免费）**

PZT 固定跑在 Mac M 系列，这是被严重低估的一层：macOS 自带的 Vision / Core Image 框架直接在端上跑，**不用下载模型、不占额外内存、结果确定**。

- 曝光：直方图分析（Core Image `CIAreaHistogram` 或解码管线里现成的像素统计）。
- 对焦/清晰度：Laplacian 方差 / 高频能量——十来行经典 CV，比让 LLM 猜"糊不糊"更可靠。
- **水平角度**：`VNDetectHorizonRequest` 直接返回地平线角度——正好补上 M3 里"模型给不准旋转几度"那个已知弱点（这本来就是个 Apple 端上 API 调用，不该外包给云端 VLM）。
- 裁切建议：`VNGenerate*SaliencyImageRequest`（显著性热图）给出主体区域。
- 人脸：`VNDetectFaceRectangles/Landmarks`（人脸数、是否有脸，人像质量的粗判）。
- 语义标签：`VNClassifyImageRequest`（场景/物体分类 → 沙滩/美食/狗...）。
- **图像嵌入（embedding）**：`VNGenerateImageFeaturePrintRequest` 端上算出一个特征向量 + 内置距离计算——**一次计算多处复用**：语义近重复、多样性挑选、检索"找日落那几张"都用得上。

一句话：M3 外包给云端 VLM 的相当一部分（水平角度、裁切建议、场景标签、相似度）其实是 Apple 端上 API。注意点：Vision/Core Image 是 Obj-C/Swift 框架，C++ core 里用要走 Objective-C++（.mm）桥接——这也是 4.1 语言决策的一个输入（若编排层是 Swift/Python，Vision 原生可用）。

**Tier 1 — 小型本地 VLM/LLM（端上，经 MLX 或 Ollama）**

真正需要"理解画面语义"的主观判断放这层：整体"是不是好片"的综合判断、构图的主观评价、语义打标签，也许还有风格→recipe。

- 怎么跑：
  - **MLX**（Apple 的数组/推理框架，吃统一内存 + GPU/ANE，M 系列上最快最原生）；`mlx-vlm` 跑视觉模型。模型 4-bit 量化后一个 2B 约 1.5-2GB 内存、7B 约 4-5GB。
  - **Ollama**（GGUF 量化、一条命令起一个本地 HTTP server，**OpenAI 兼容接口**）。这是**集成最省事**的路：M3 已经在打 HTTP API（Gemini/Claude），把 endpoint 指向 `localhost:11434` 是很小的改动。
- 候选模型：Moondream2（~2B，专为快速 VQA/描述设计）、Qwen2-VL/Qwen2.5-VL（2B/7B，**中文强**）、MiniCPM-V、Florence-2（小、擅长检测/描述/区域任务）。
- **跟 M3 的复用点（关键卖点）**：`core/ai/ai.h` 现在是 `Provider {Claude, Gemini}` + 可注入的 `HttpPostFn`。加一个 `Provider::Local`（或注入一个指向本地 endpoint 的 `HttpPostFn`）就把本地模型接进来了——**本地模型"只是又一个 provider"**，异步 worker 队列、落库、去重逻辑全不用动。这正是 M3 那套 provider 抽象 + 依赖注入当初就留好的口子。

**Tier 2 — 云端 LLM（Gemini/Claude，仅限低频任务）**

只留给"一批一次、且要强推理/强中文创作"的任务：写文案、也许最终的创意策展。因为一批只调一次，**成本天然有界**，这直接回答了用户"token 扛不住"的顾虑——不是把所有东西本地化，而是**把逐张的高频任务下沉到 Tier 0/1，只把一次性的创作留在 Tier 2**。

补充：写文案未必非云端不可——Qwen 系列（Qwen2.5-7B / Qwen2-VL-7B）中文很强，本地也能出还不错的中文文案。所以"文案留云端"是**成本考量下的默认**，不是能力所限；用户若把"照片不出机"设成硬约束，可以整条换成本地 Qwen，代价是文案质量和速度打点折扣。这条是可配置的取舍，不是死规矩。

### 5.3 分层如何映射到 M4 流水线

| 流水线步骤 | 落在哪层 | 说明 |
|---|---|---|
| 曝光 / 对焦评分 | Tier 0 | 经典 CV，确定、免费、比 LLM 猜得准 |
| 水平角度 / 裁切建议 | Tier 0 | Apple Vision 直接给 |
| 构图 / 整体"是否好片"综合判断 | Tier 1 | 需要语义理解的主观判断 |
| 语义标签（沙滩/美食…） | Tier 0 或 Tier 1 | `VNClassifyImage` 起步，不够再上小 VLM |
| 去重 | 已本地（dHash） | 可选：用 Tier 0 的 feature print 升级到语义近重复 |
| 挑 9 张（多样性） | 算法 + Tier 0 embedding | 见第七节，未必要 LLM |
| 风格 → recipe | Tier 1 或 规则库 | 小结构化输出；可能预设 LUT 库 + 关键词匹配比 LLM 更稳 |
| 写文案 | Tier 2（或本地 Qwen） | 低频、强中文创作 |

## 六、"挑 9 张"策展子问题（值得单独想）

朴素 top-9 by `overall_score` 会挑出 9 张几乎相同的照片（同一个日落连拍全在前排），完全讲不成"一天的故事"。这是个真问题，几种方向：

- **嵌入聚类**：用 Tier 0 的 feature print 把候选聚成 K 类，每类挑分数最高的代表 → 保证覆盖度/多样性。
- **时间铺开**：用 `captured_at` 把 9 张沿一天的时间线摊开（但 WhatsApp 剥 EXIF 时这个信号可能没有，得靠像素通道拿原图）。
- **约束优化**：在"总分最大"的目标下加"多样性/时间分布"约束。
- **LLM 策展**：把 top-N 缩略图丢给多模态模型让它选。质量不确定，作为增强而非基线。

跟 M3 的接口天然咬合：候选来自 `passes_gate` 过滤 + 排除 reject/dup 标签（`images_with_tag`），排序基准是 `overall_score`，去重分组（`DuplicateGroup`）本身就是一种"这些太像"的信号可以直接用于多样性。**基线用算法策展（确定、免费），LLM 策展作为可选增强**。

## 七、与 M3 遗留问题的交叉

M4 让 M3 "风险与待确认问题"里的好几条从"悬着"变成"该在 M4 里回答"，说明两者确实"互为增强"：

- **M3 的"自动介入触发条件"**（`pzt new`/`rescan` 自动排队评分？空闲自动补跑？）——**M4 就是自动介入的语境**，agent 收到照片自动触发送评。M4 会回答/吸收这个 M3 悬案。
- **M3 的"自动打标签"**（完全没讨论）——M4 的语义标签（沙滩/美食）就是它，被吸收。
- **M3 的云端 vs 本地取舍**——第五节的本地分层设计直接正面回应。
- **M3 的"修正建议如何真正应用"（recipe 缺几何变换）**——M4 的"统一风格"要套 recipe；若"风格"含裁切/水平矫正，会**重新激活这个几何变换缺口**（`VersionParams` 目前只有色调/白平衡），需要评估是否要给 recipe/渲染管线补一层几何变换。是一个可能的前置依赖。
- **去重的 captured_at 依赖**——WhatsApp 剥 EXIF，M4 摄入路径下去重的时间聚类会退化；这也是 4.2"像素通道走原图"的动机之一。
- **M3 的异步 vs 阻塞（dedup）**——M4 批处理场景下，dedup 冻结几十秒更不可接受，可能把"给 dedup 做异步版"从 M3 的悬案推成 M4 的实际需求。

## 八、分阶段落地设想

跟 Roadmap"先用简单规则跑通编排链路"一致，从最小闭环起步，逐层加复杂度：

1. **v0 骨架（规则驱动，无 IM、无本地模型）**：监听一个文件夹 → CLI 触发一次"agent run" → 复用 core 跑 送评（先仍用云端）→ 去重 → 算法挑 N 张 → 导出到成品目录。目的：把**编排状态机 + core 复用**跑通，证明链路。
2. **v1 本地化评估层**：把 Tier 0（经典 CV + Apple Vision）和 Tier 1（Ollama/MLX 小 VLM 作为新 provider）接进送评，把逐张成本压下去。
3. **v2 IM 传输**：接 WhatsApp（控制通道），像素走文件夹/照片库；加进度回报和基础对话确认。
4. **v3 策展/文案增强**：多样性策展、风格→recipe、云端文案、对话式微调。

## 九、待拍板 / 待确认问题（尚无结论，需用户拍板）

1. **编排层语言 / 调 core 的边界**（4.1）——头号岔路。C++ 单一工具链 vs Python/Swift 靠近集成生态。牵涉长期维护成本和 `core` 要不要补 headless 入口。
2. **主要用户是谁**——是"个人发朋友圈"，还是也含"摄影师给客户交付初筛"？决定哪些用例是核心、策展/交付形态怎么设计。（PZT 的 RAW/recipe/有序标签更像准专业向，但朋友圈用例是纯个人向，两者对 v1 的重心影响不同。）
3. **"照片不出机"是硬约束还是偏好**——决定本地模型要做到多彻底、文案能不能上云。
4. **WhatsApp 接入方式**（4.2）——官方 Business API（合规、有成本/审批）vs 非官方库（好用、有封号风险）；以及要不要采用"控制通道 vs 像素通道分离"。
5. **v1 野心边界**——是先做"纯粗筛交付"最小闭环，还是一上来就奔完整的朋友圈端到端？（本文档倾向前者。）
6. **风格是否含几何变换**——若"统一风格"要裁切/水平矫正，需先决定要不要给 recipe 管线补几何变换能力（激活 M3 遗留缺口），这会显著加大 recipe 侧工作量。
