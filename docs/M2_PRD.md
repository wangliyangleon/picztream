# PicZTream (PZT) Milestone 2 产品需求文档（RAW 支持）

## 背景与问题陈述

`docs/Roadmap.md` 定义的双层流水线到 M1 为止只验证了一半：culling（秒切浏览）和 processing（风格化处理）都只在纯 JPEG 场景下跑通，`core/decode` 至今没有真正碰过 RAW 文件。M2 要打通另一半——接入 LibRaw，让项目可以直接导入相机原始 RAW 文件（徕卡 DNG、富士 X-Trans RAF），且严格遵循"按需触发"原则：浏览阶段绝大多数操作不触发 RAW 解码，只有真正要产出最终成品（导出）时才付出这个代价。

这一步的验证目标不是"RAW 处理效果好不好"（那需要真正的调色设计工作，不是这次的重点），而是验证"RAW 输入能不能无缝嵌进已经跑通的选片+风格化工作流，且不拖慢日常浏览"这个架构假设是否成立。

## M2 目标

* 项目可以导入包含 DNG / RAF 文件的文件夹，浏览体验（切图延迟、打标签、筛选）与纯 JPEG 项目无感知差异
* RAW 来源的图片可以像 JPEG 图片一样应用 M1 已有的 `recipe` 风格系统，浏览时看到预览效果，且预览的色彩/影调跟真正导出时的效果高度一致（不是完全不相关的两套渲染）
* `pzt export` 导出 RAW 来源的图片时，始终基于 LibRaw 真正解码出的结果（无论是否应用了风格），产出可直接查看使用的 JPEG

## 非目标

* 不做 RAW 专属的调整维度（比如 highlight recovery、去马赛克算法选择、色差校正）——`recipe` 的参数集合维持 M1 已定的高光/暗光/白平衡偏移，不为 RAW 新增专属旋钮
* 不做 RAW 专属的自动校正算法（曝光/白平衡自动优化属于 M3"AI 辅助"范畴的自动粗修，不是 M2 的按需解码本身）
* 不做除 DNG / RAF 之外的格式（CR2/CR3、NEF、ARW 等）——留到真正有对应器材需求时再加，但 Eng Design 阶段的格式判断逻辑要为后续扩展留好口子，不写死只认这两种扩展名的分散判断
* 不新增"标记精修"这类独立于现有标签/`recipe` 系统之外的专用机制——沿用 M1 已有的 `recipe` 应用（`r` 菜单）和 M0 已有的自定义标签，用户如果自己想追踪"哪些图片打算精修"，自建一个标签筛选即可，不是 core 需要理解的概念
* 不做 sidecar（`.xmp` 等）读写，不与 Lightroom / Capture One 等外部工具的编辑记录互通
* 不做色彩管理（ICC）方面的新工作——延续 M0/M1"信任系统默认色彩匹配"的简化假设；但 LibRaw 解码路径不经过 CoreGraphics 的 ColorSync 自动处理，这个假设的风险边界在 RAW 路径上跟 JPEG 路径不同，需要在 Eng Design 阶段重新审视（见"风险与待确认问题"）

## 核心用户故事

* 用户可以把一个混有 DNG / RAF 文件的文件夹导入为项目；`pzt new`/`pzt rescan` 处理 RAW 文件时需要一点准备时间（生成预览缓存），过程中能看到进度提示，不会误以为程序卡死
* 准备完成后，浏览体验（切图延迟、打标签、筛选、`key-to-render` 延迟）跟纯 JPEG 项目没有可感知差异
* 用户可以在信息栏看到当前图片的来源类型（RAW / JPEG），不需要额外查看文件属性
* 用户可以对 RAW 来源的图片打标签、应用 `recipe` 风格，操作方式和交互反馈跟 JPEG 图片完全一致；预览效果基于 `new`/`rescan` 时生成的 RAW 预览缓存渲染，色彩/影调跟真正导出时的效果一致（都是 LibRaw 解码的产物，只是分辨率不同），不会出现"预览看到一个样子、导出出来是另一个样子"的落差
* 用户导出某个标签下的图片时：RAW 来源的图片无论是否应用风格都会得到一个基于 LibRaw 真正解码出的、可直接查看使用的 JPEG（RAW 文件本身大多数下游工具打不开，导出不应该把这个负担留给用户）
* 用户如果某张照片是 RAW+JPEG 同时拍摄（同名不同扩展名），系统直接把它当 RAW 处理，忽略那份 JPEG——不会在项目里重复计数、重复出现两条浏览记录，也不需要用户操心"我这张照片算 RAW 还是算 JPEG"

## 功能需求

### 项目扫描与格式识别

* `pzt new` / `pzt rescan` 的扫描范围从"只认 `.jpg`/`.jpeg`"扩展为同时识别 `.dng`/`.raf`（大小写不敏感），与现有 JPEG 扫描逻辑并列，不是替换
* 同一文件夹内，文件名主干（不含扩展名）相同的 RAW 文件与 JPEG 文件（例如 `IMG_001.RAF` 与 `IMG_001.JPG`）同时存在时，只认 RAW，JPEG 那份被忽略、不生成 `images` 记录——RAW 才是"真正的原始数据"，一旦决定预览也要基于 RAW 解码（见下），继续留着一条指向伴侣 JPEG 的记录只会重新引入"预览和导出用两份不同素材"的落差，不如从一开始就不追踪它
* 每张图片记录自己的来源类型：JPEG / RAW。信息栏展示这个状态，具体文案位置对齐现有信息栏"尺寸:"/"风格:"这类字段的展示风格

### RAW 预览缓存（`new`/`rescan` 时的一次性成本）

* `pzt new` / `pzt rescan` 扫描到 RAW 文件后，会立刻用 LibRaw 做一次**降分辨率**解码（不是全量解码），把结果编码成 JPEG 存进 PZT 自己的数据目录，作为这张图之后所有预览用途的素材来源——这是 M2 的核心设计调整：预览不再依赖相机内嵌的缩略图，而是基于"真正解码过 RAW 数据、只是分辨率低一些"的结果，这样预览看到的色彩/影调才会跟导出时高度一致，不会出现"预览是一个样子、导出是另一个样子"这种容易让人诧异的落差（真机验证 increment 2 时就实测遇到过：某些相机的内嵌预览可能跟 RAW 解码结果观感差异很大）
* 这一步不是免费的——降分辨率解码仍然要跑真正的 LibRaw 去马赛克管线，只是比全量解码快得多。命令执行期间需要给用户看得见的进度提示（"正在处理第 X/N 张"这类），不能让用户以为程序卡死了；执行完之后，浏览这个项目的体验（切图延迟等）才恢复到跟纯 JPEG 项目一样快，因为预览这时候已经变成"解码一张普通 JPEG"这个已经验证过的路径
* 缓存文件不写进用户的照片文件夹——延续 M0 起"绝不修改/污染用户照片目录"的原则，缓存连同其它 PZT 状态一起放在 PZT 自己的数据目录下，用户如果直接浏览照片文件夹，看不到任何 PZT 生成的痕迹
* 缓存丢失（比如用户手动清过 PZT 的数据目录）时浏览仍然要能工作，只是临时退化到用相机内嵌预览这个更快但可能不够准的兜底路径，不会导致某张图直接打不开

### 浏览与选片（culling，延迟敏感路径）

* RAW 来源图片的浏览预览，正常情况下直接解码上一节生成的预览缓存文件；缓存缺失时退化到"提取 RAW 文件内嵌的相机 ISP 预览 JPEG"这条兜底路径（对齐 `docs/Roadmap.md` 描述的"读取 RAW 文件末尾内嵌高质量 JPEG，绕过 RAW 解码"），两条路径都不触发 LibRaw 全量解码
* 两种情况读到的预览都喂给现有的 `core::decode::DecodedImage` / 预取缓存 / Kitty 渲染整条链路，跟现有纯 JPEG 路径复用同一套下游代码，不需要为 RAW 来源单独设计一套预览管线
* `h`/`l`/`j`/`k` 导航、`space` 打标签、`x` 废片、`g` 筛选等既有交互，对 RAW 来源图片的行为跟 JPEG 图片完全一致，不需要新的按键或菜单分支
* 延迟目标沿用 M0/M1 已确认的"key-to-render 目标 100ms 以内、本地 SSD + 预取命中"标准——这个预算适用于浏览阶段本身（解码预览缓存文件），不适用于 `new`/`rescan` 时一次性生成缓存这一步（那是批处理，见上一节）

### 风格化预览（复用 M1 `recipe` 系统，不新增机制）

* `r` 菜单（应用/创建/删除 `recipe` version）对 RAW 来源图片和 JPEG 图片一视同仁，选择、切换、软删除的交互完全复用 M1 已有实现，不新增按键或分支
* 预览渲染出的效果，是"RAW 预览缓存解码结果 + `recipe` 处理"，跟导出时"RAW 全量解码结果 + `recipe` 处理"用的是同一条色彩管线、同样的白平衡/色彩矩阵/gamma，只是分辨率不同——这是这次调整后预览和导出观感一致的直接来源，不再是刻意接受的近似

### 导出（processing 阶段，唯一真正触发 LibRaw 全量解码的入口）

`pzt export <project_name> <tag_name> <output_folder>` 按图片来源类型和是否应用了 `recipe` 分情况处理：

| 来源类型 | 是否应用 recipe | 导出行为 |
|---|---|---|
| JPEG | 否 | 原样复制/软链 JPEG 字节（M0/M1 已有行为，不变） |
| JPEG | 是 | 解码 JPEG → 渲染 → 编码（M1 已有行为，不变） |
| RAW | 否 | LibRaw 全量解码 → 不做任何调整（中性）→ 编码成 JPEG |
| RAW | 是 | LibRaw 全量解码 → 渲染 → 编码成 JPEG |

* RAW 图片导出永远落地成全量解码的 JPEG——RAW 文件本身大多数下游工具（发朋友圈、随手分享）打不开，导出的产品目标是给用户一个可以直接用的成品；`new`/`rescan` 时生成的预览缓存分辨率不够，不能拿来冒充导出结果，导出必须重新做一次全量解码
* `--link` 模式下，需要真正解码烘焙的图片没有"原始字节"可以软链，只能落地成真实文件，这是 M1 已经确立的限制，直接延伸到 RAW 路径
* 因为全量解码只发生在 `pzt export` 这个一次性批处理命令内部，不发生在交互浏览循环里，不需要遵守 100ms 的 key-to-render 延迟预算——沿用 M1 已经确立的"导出可以用 `jthread` 并行、不需要常驻后台队列"的模型即可

## 非功能需求

* **浏览延迟：** 沿用 M0/M1 标准，RAW 来源图片的浏览体验（切图、打标签、筛选）与 JPEG 图片无可感知差异，配合已有的 `key-to-render` 延迟日志验证——前提是 `new`/`rescan` 时的预览缓存生成已经完成
* **`new`/`rescan` 耗时与反馈：** 扫描到 RAW 文件时命令本身会变慢（生成预览缓存需要真正跑一遍 LibRaw 降分辨率解码），必须有进度提示，不能让用户误以为程序卡死；这条不适用于纯 JPEG 项目，纯 JPEG 场景的 `new`/`rescan` 耗时不变
* **导出正确性：** RAW 图片导出（无论是否应用风格）必须体现真正基于 LibRaw 全量解码后的效果，不能用预览缓存或内嵌预览的分辨率/结果冒充
* **数据持久性：** 图片的来源类型需要持久化；RAW 图片的预览缓存文件路径也需要持久化，重新打开项目、`rescan` 之后缓存能被正确复用，不会每次都重新生成
* **平台支持：** 沿用 M0/M1，仅支持 macOS M 系列芯片配合 Ghostty 终端与 Kitty 图像协议；新增的 LibRaw 依赖需要确认在这个平台上通过 Homebrew 可正常安装、链接
* **不污染用户数据：** 延续 M0 起"不修改/不写入用户照片目录"的原则，RAW 预览缓存只能写进 PZT 自己的数据目录，不能以任何形式（含隐藏文件）写进用户的照片文件夹

## 技术方案概要

（细节留给 Eng Design，这里只定方向）

* LibRaw 作为新的第三方依赖引入，只出现在 `core`（新的 `core/raw/` 模块），不泄漏到 `cli`，延续项目一贯的分层约定
* 预览缓存生成：`new`/`rescan` 时用 LibRaw 的降分辨率解码模式（而不是全量解码）为每张 RAW 图片生成一份缓存 JPEG，具体存放路径、命名规则、生命周期（项目删除/图片被 `rescan` 清理时缓存要不要跟着清）留给 Eng Design，但必须落在 PZT 自己的数据目录下，不进用户照片文件夹
* Processing 路径（真正导出）：LibRaw 全量解码产出 8-bit sRGB，直接复用现有 `core::color`/`recipe::render` 整条管线，不新增色彩代码（已在 Eng Design 阶段确认）
* 数据模型层面需要新增字段表达"图片来源类型"与"RAW 预览缓存路径"，具体 schema 设计留给 Eng Design

## 验收标准

* [ ] 可以创建一个包含 DNG 和 RAF 文件（含至少一组同名 RAW+JPEG，验证 JPEG 那份被正确忽略、不重复计数）的项目，扫描正确识别所有文件
* [ ] `pzt new`/`pzt rescan` 处理 RAW 文件时有可见的进度提示，不是静默等待
* [ ] 预览缓存生成完成后，浏览这个项目的全过程主观感受与浏览等量纯 JPEG 项目无可感知差异，配合 `key-to-render` 延迟日志验证
* [ ] 信息栏正确显示每张图片的来源类型（RAW / JPEG）
* [ ] 可以对 RAW 来源图片打标签、应用 `recipe`，交互行为与 JPEG 图片一致，预览能看到风格效果，且预览色彩跟导出结果肉眼一致（不是内嵌预览那种可能完全不搭的效果）
* [ ] `pzt export` 对 RAW 图片（无论是否应用 `recipe`）导出的都是真正经过 LibRaw 全量解码的 JPEG 成品，未应用 `recipe` 时是中性渲染，应用了时体现处理效果
* [ ] 用户照片文件夹里找不到任何 PZT 写入的文件（缓存确实落在 PZT 自己的数据目录下）
* [ ] 核心逻辑（格式识别、缓存生成、导出分支路由）具备基本单元测试覆盖

## 风险与待确认问题

* **LibRaw 解码的白平衡/色彩基线**：这个决定同时定义"预览缓存长什么样"和"RAW 图片不应用 recipe 时导出的中性 JPEG 长什么样"，两者必须用同一套基线才能保持一致。倾向于让 LibRaw 解码使用相机 as-shot 白平衡作为基线（而不是完全不做白平衡修正的原始传感器读数，那样会明显偏色，不能算"中性"），`recipe` 的 `wb_shift_r/b` 在此基础上叠加。这是本文档起草时的建议，不是拍板决定，具体数值处理方式需要 Eng Design 阶段结合真机测试确定
* **色彩管理边界发生变化**：LibRaw 输出的线性 RGB 不经过 CoreGraphics ColorSync 自动匹配，M0/M1"信任系统默认处理"的假设在这条路径上不成立，需要在 Eng Design 阶段明确 LibRaw 输出到最终 8-bit JPEG 之间的 gamma/色域转换是否正确，避免看起来能跑通但颜色是错的
* **预览缓存缺失的降级路径**：设计上要求缓存缺失时退化到内嵌预览提取，这条路径本身依赖"内嵌预览是 JPEG 格式"这个假设（真机验证过徕卡/富士两种测试机型成立），如果真机测试中遇到内嵌预览也缺失/损坏的例外，进一步的降级行为（报错？临时现场跑一次降分辨率解码？）留到真正遇到实例时再处理
* **`new`/`rescan` 处理大量 RAW 文件的总耗时**：单张降分辨率解码耗时因机型差异很大（真机实测徕卡 DNG ~1 秒、富士 RAF 反而只要 ~0.1 秒），几百张的真实项目累计耗时需要 Eng Design 阶段给出量级评估，判断要不要用多线程并行处理多张图片来缩短总等待时间

## 未来考虑

* 更多 RAW 格式支持（CR2/CR3、NEF、ARW 等），等有对应器材需求时再加，Eng Design 阶段的格式判断逻辑需要为此留好扩展口子
* RAW 专属调整参数（highlight recovery、色差校正等），等真机使用中发现 M1 现有的高光/暗光/白平衡偏移不够用再考虑
* **导出时还原相机机内风格（胶片模拟/黑白模式）**：increment 2 真机验证时发现，两张徕卡样张的内嵌预览是黑白的（拍摄时相机开了黑白风格），但 LibRaw 通用去马赛克管线吐出来永远是彩色。查过 LibRaw 0.22.1 头文件：徕卡完全没有对应的 makernotes 解析（`libraw_leica_makernotes_t` 这个结构体根本不存在），技术上做不到；富士倒是有 `libraw_fuji_info_t.FilmMode` 这个原始数值字段，但 LibRaw 不解释它是哪个模式，且知道模式本身也不代表能还原它的调色曲线。这次改成"预览也走 LibRaw 解码"之后，这个问题的表现形式变了——预览和导出至少不会再互相矛盾（都是彩色），但都跟相机原本想呈现的黑白效果不一致，这个落差依然存在，仍然不在 M2 范围内解决
