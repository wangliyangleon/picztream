# PicZTream (PZT) Milestone 2 产品需求文档（RAW 支持）

## 背景与问题陈述

`docs/Roadmap.md` 定义的双层流水线到 M1 为止只验证了一半：culling（秒切浏览）和 processing（风格化处理）都只在纯 JPEG 场景下跑通，`core/decode` 至今没有真正碰过 RAW 文件。M2 要打通另一半——接入 LibRaw，让项目可以直接导入相机原始 RAW 文件（徕卡 DNG、富士 X-Trans RAF），且严格遵循"按需触发"原则：浏览阶段绝大多数操作不触发 RAW 解码，只有真正要产出最终成品（导出）时才付出这个代价。

这一步的验证目标不是"RAW 处理效果好不好"（那需要真正的调色设计工作，不是这次的重点），而是验证"RAW 输入能不能无缝嵌进已经跑通的选片+风格化工作流，且不拖慢日常浏览"这个架构假设是否成立。

## M2 目标

* 项目可以导入包含 DNG / RAF 文件的文件夹，浏览体验（切图延迟、打标签、筛选）与纯 JPEG 项目无感知差异
* RAW 来源的图片可以像 JPEG 图片一样应用 M1 已有的 `recipe` 风格系统，浏览时看到预览效果
* `pzt export` 导出图片时，纯 RAW 图片始终解码成可直接查看使用的 JPEG（无论是否应用了风格）；RAW+JPEG 配对图片如果没有应用风格，直接导出原始 JPEG 伴侣文件，只有应用了风格才触发 LibRaw 全量解码烘焙

## 非目标

* 不做 RAW 专属的调整维度（比如 highlight recovery、去马赛克算法选择、色差校正）——`recipe` 的参数集合维持 M1 已定的高光/暗光/白平衡偏移，不为 RAW 新增专属旋钮
* 不做 RAW 专属的自动校正算法（曝光/白平衡自动优化属于 M3"AI 辅助"范畴的自动粗修，不是 M2 的按需解码本身）
* 不做除 DNG / RAF 之外的格式（CR2/CR3、NEF、ARW 等）——留到真正有对应器材需求时再加，但 Eng Design 阶段的格式判断逻辑要为后续扩展留好口子，不写死只认这两种扩展名的分散判断
* 不新增"标记精修"这类独立于现有标签/`recipe` 系统之外的专用机制——沿用 M1 已有的 `recipe` 应用（`r` 菜单）和 M0 已有的自定义标签，用户如果自己想追踪"哪些图片打算精修"，自建一个标签筛选即可，不是 core 需要理解的概念
* 不做 sidecar（`.xmp` 等）读写，不与 Lightroom / Capture One 等外部工具的编辑记录互通
* 不做色彩管理（ICC）方面的新工作——延续 M0/M1"信任系统默认色彩匹配"的简化假设；但 LibRaw 解码路径不经过 CoreGraphics 的 ColorSync 自动处理，这个假设的风险边界在 RAW 路径上跟 JPEG 路径不同，需要在 Eng Design 阶段重新审视（见"风险与待确认问题"）

## 核心用户故事

* 用户可以把一个混有 DNG / RAF 文件的文件夹导入为项目，浏览体验（切图延迟、打标签、筛选、`key-to-render` 延迟）跟纯 JPEG 项目没有可感知差异
* 用户可以在信息栏看到当前图片的来源类型（RAW / JPEG / RAW&JPEG），不需要额外查看文件属性
* 用户可以对 RAW 来源的图片打标签、应用 `recipe` 风格，操作方式和交互反馈跟 JPEG 图片完全一致；预览效果基于内嵌/伴侣 JPEG 渲染，不是真正的 RAW 解码结果，但足够用户判断风格选择是否合适
* 用户导出某个标签下的图片时：纯 RAW 图片无论是否应用风格都会得到一个可直接查看使用的 JPEG（RAW 文件本身大多数下游工具打不开，导出不应该把这个负担留给用户）；RAW+JPEG 配对图片如果没有应用风格，直接拿到相机原生 JPEG，不触发任何解码；只有配对图片应用了风格，才会触发 LibRaw 全量解码烘焙出最终效果
* 用户如果某张照片是 RAW+JPEG 同时拍摄（同名不同扩展名），系统自动识别为同一张逻辑图片，不会在项目里重复计数、重复出现两条浏览记录

## 功能需求

### 项目扫描与格式识别

* `pzt new` / `pzt rescan` 的扫描范围从"只认 `.jpg`/`.jpeg`"扩展为同时识别 `.dng`/`.raf`（大小写不敏感），与现有 JPEG 扫描逻辑并列，不是替换
* 同一文件夹内，文件名主干（不含扩展名）相同的 RAW 文件与 JPEG 文件（例如 `IMG_001.RAF` 与 `IMG_001.JPG`）识别为同一张逻辑图片，只生成一条 `images` 记录，不重复计数
* 每张图片记录自己的来源类型：纯 JPEG / 纯 RAW / RAW+JPEG 配对。信息栏展示这个状态（"RAW" / "JPEG" / "RAW&JPEG"），具体文案位置对齐现有信息栏"尺寸:"/"风格:"这类字段的展示风格
* 对于 RAW+JPEG 配对的图片，是否需要真正碰 RAW 数据取决于有没有应用 `recipe`：没有应用（Origin）时浏览预览和导出都直接用 JPEG 伴侣文件；应用了风格之后，导出才会转向以 RAW 为准解码烘焙（浏览预览仍然用 JPEG，见"浏览与选片"一节）

### 浏览与选片（culling，延迟敏感路径）

* 纯 RAW 图片的浏览预览，走"提取 RAW 文件内嵌的相机 ISP 预览 JPEG"这条路径（对齐 `docs/Roadmap.md` 描述的"读取 RAW 文件末尾内嵌高质量 JPEG，绕过 RAW 解码"），不触发 LibRaw 解码
* RAW+JPEG 配对图片的浏览预览，直接解码 JPEG 伴侣文件本身，连"从 RAW 里提取内嵌预览"这一步都不需要——伴侣文件就是相机产出的完整 JPEG，比内嵌预览更直接可用，也省掉一次提取开销
* 两种情况提取/读到的预览都喂给现有的 `core::decode::DecodedImage` / 预取缓存 / Kitty 渲染整条链路，跟现有纯 JPEG 路径复用同一套下游代码，不需要为 RAW 来源单独设计一套预览管线
* `h`/`l`/`j`/`k` 导航、`space` 打标签、`x` 废片、`g` 筛选等既有交互，对 RAW 来源图片的行为跟 JPEG 图片完全一致，不需要新的按键或菜单分支
* 延迟目标沿用 M0/M1 已确认的"key-to-render 目标 100ms 以内、本地 SSD + 预取命中"标准，RAW 来源图片的预览提取步骤需要在这个预算内完成

### 风格化预览（复用 M1 `recipe` 系统，不新增机制）

* `r` 菜单（应用/创建/删除 `recipe` version）对 RAW 来源图片和 JPEG 图片一视同仁，选择、切换、软删除的交互完全复用 M1 已有实现，不新增按键或分支
* 预览渲染出的效果，是"内嵌/伴侣 JPEG 解码结果 + `recipe` 处理"的近似效果，不是"真正 RAW 解码 + `recipe` 处理"的精确效果——这个近似的取舍是 M2 刻意的范围裁剪：既然浏览阶段不触发 LibRaw，预览就只能基于已经拿到手的 JPEG 数据渲染，用户凭这个近似效果判断"要不要标记导出、选哪个风格"已经够用，真正精确的效果要导出后才能看到

### 导出（processing 阶段，唯一真正触发 LibRaw 解码的入口）

`pzt export <project_name> <tag_name> <output_folder>` 按图片来源类型和是否应用了 `recipe` 分情况处理：

| 来源类型 | 是否应用 recipe | 导出行为 |
|---|---|---|
| 纯 JPEG | 否 | 原样复制/软链 JPEG 字节（M0/M1 已有行为，不变） |
| 纯 JPEG | 是 | 解码 JPEG → 渲染 → 编码（M1 已有行为，不变） |
| 纯 RAW | 否 | LibRaw 全量解码 → 不做任何调整（中性）→ 编码成 JPEG |
| 纯 RAW | 是 | LibRaw 全量解码 → 渲染 → 编码成 JPEG |
| RAW+JPEG 配对 | 否 | 原样复制/软链 **JPEG 伴侣文件**，不碰 RAW，不解码 |
| RAW+JPEG 配对 | 是 | LibRaw 全量解码 **RAW** → 渲染 → 编码成 JPEG |

* 纯 RAW 图片导出永远落地成 JPEG——RAW 文件本身大多数下游工具（发朋友圈、随手分享）打不开，导出的产品目标是给用户一个可以直接用的成品，因此纯 RAW 图片无论是否应用风格都会触发 LibRaw 全量解码，"按需触发"在这里的含义是"按需触发解码"而不是"按需触发导出方式"
* RAW+JPEG 配对图片则严格按需触发：没有应用风格时相机自己出的 JPEG 已经是现成的可用成品，直接给用户，不解码不重新编码；应用了风格才需要用更高质量的 RAW 数据重新烘焙
* `--link` 模式下，需要真正解码烘焙的图片没有"原始字节"可以软链，只能落地成真实文件，这是 M1 已经确立的限制，直接延伸到 RAW 路径
* 因为解码只发生在 `pzt export` 这个一次性批处理命令内部，不发生在交互浏览循环里，不需要遵守 100ms 的 key-to-render 延迟预算——这条延迟约束天然不适用于 M2 新增的 RAW 全量解码路径，沿用 M1 已经确立的"导出可以用 `jthread` 并行、不需要常驻后台队列"的模型即可

## 非功能需求

* **浏览延迟：** 沿用 M0/M1 标准，RAW 来源图片的浏览体验（切图、打标签、筛选）与 JPEG 图片无可感知差异，配合已有的 `key-to-render` 延迟日志验证
* **导出正确性：** 纯 RAW 图片导出（无论是否应用风格）必须体现真正基于 LibRaw 解码后的效果，不能用内嵌预览的解码结果冒充；RAW+JPEG 配对图片没有应用风格时，导出必须是 JPEG 伴侣文件字节的精确拷贝（不解码不重新编码）；应用了风格的配对图片，导出必须体现真正基于 RAW 数据解码后的处理效果
* **数据持久性：** 图片的来源类型、RAW/JPEG 配对关系需要持久化，重新打开项目、`rescan` 之后配对关系保持稳定
* **平台支持：** 沿用 M0/M1，仅支持 macOS M 系列芯片配合 Ghostty 终端与 Kitty 图像协议；新增的 LibRaw 依赖需要确认在这个平台上通过 Homebrew 可正常安装、链接

## 技术方案概要

（细节留给 Eng Design，这里只定方向）

* LibRaw 作为新的第三方依赖引入，只出现在 `core`（大概率是新的 `core/raw/` 模块或 `core/decode` 的增补），不泄漏到 `cli`，延续项目一贯的分层约定
* Culling 路径：从 RAW 文件提取内嵌预览 JPEG 这一步，具体走 LibRaw 自带的缩略图提取接口，还是手写协议级偏移解析，是 Eng Design 阶段的实现选型问题，不是产品需求；产品需求只要求这一步不触发完整的 LibRaw 解马赛克，且性能上跟直接读一个 JPEG 文件同量级
* Processing 路径：LibRaw 全量解码产出的线性数据，如何接入现有 `core::color`（目前只认 8-bit `DecodedImage`）的问题，留给 Eng Design 决定是转换成现有结构复用整条 `recipe::render` 管线，还是新增一条 16-bit 专用路径
* 数据模型层面需要新增字段/表来表达"图片来源类型"与"RAW↔JPEG 配对关系"，具体 schema 设计留给 Eng Design

## 验收标准

* [ ] 可以创建一个包含 DNG 和 RAF 文件（含至少一组同名 RAW+JPEG 配对）的项目，扫描正确识别所有文件，配对文件不重复计数
* [ ] 浏览这个项目的全过程主观感受与浏览等量纯 JPEG 项目无可感知差异，配合 `key-to-render` 延迟日志验证
* [ ] 信息栏正确显示每张图片的来源类型（RAW / JPEG / RAW&JPEG）
* [ ] 可以对 RAW 来源图片打标签、应用 `recipe`，交互行为与 JPEG 图片一致，预览能看到风格效果
* [ ] `pzt export` 对纯 RAW 图片（无论是否应用 `recipe`）导出的都是真正经过 LibRaw 解码的 JPEG 成品，未应用 `recipe` 时是中性渲染，应用了时体现处理效果
* [ ] `pzt export` 对 RAW+JPEG 配对且未应用 `recipe` 的图片，导出的是 JPEG 伴侣文件的精确字节拷贝，不触发 LibRaw
* [ ] `pzt export` 对 RAW+JPEG 配对且应用了 `recipe` 的图片，导出文件体现真正基于 LibRaw 解码 RAW 数据后的处理效果（不是用 JPEG 伴侣文件冒充）
* [ ] 核心逻辑（格式识别、配对逻辑、LibRaw 解码封装、导出分支路由）具备基本单元测试覆盖

## 风险与待确认问题

* **LibRaw 解码的白平衡/色彩基线**：这个决定现在直接定义"纯 RAW 图片不应用 recipe 时导出的中性 JPEG 长什么样"，不只是预览一致性的锦上添花。倾向于让 LibRaw 解码使用相机 as-shot 白平衡作为基线（而不是完全不做白平衡修正的原始传感器读数，那样会明显偏色，不能算"中性"），`recipe` 的 `wb_shift_r/b` 在此基础上叠加——这样纯 RAW 图片导出的默认效果才会接近内嵌/伴侣 JPEG 预览看到的观感，符合用户预期。这是本文档起草时的建议，不是拍板决定，具体数值处理方式需要 Eng Design 阶段结合真机测试确定
* **色彩管理边界发生变化**：LibRaw 输出的线性 RGB 不经过 CoreGraphics ColorSync 自动匹配，M0/M1"信任系统默认处理"的假设在这条路径上不成立，需要在 Eng Design 阶段明确 LibRaw 输出到最终 8-bit JPEG 之间的 gamma/色域转换是否正确，避免看起来能跑通但颜色是错的
* **内嵌预览缺失的边缘情况**：假设 DNG/RAF 文件都带可用的内嵌预览 JPEG（行业惯例），如果真机测试中遇到例外（内嵌预览缺失或损坏），降级行为（报错跳过？退化成完整 LibRaw 解码出预览？）留到真正遇到实例时再处理，不预先设计
* **LibRaw 解码性能未知**：富士 X-Trans 阵列的去马赛克通常比拜耳阵列更慢，目前没有实测数据；Eng Design 阶段需要类似 M1 Phase 0 spike 的方式先测一次真实的 DNG/RAF 全量解码耗时，再决定 `pzt export` 是否需要给用户加一个"正在处理 X/N"的进度反馈（当前 M1 导出没有进度提示，因为 JPEG 烘焙够快；RAW 解码如果明显更慢，这可能是导出体验上需要补的一环）
* **RAW+JPEG 配对的判定规则边界**：同名不同扩展名是最简单的配对规则，但如果同一文件夹里出现"两个 RAW 格式同名"（比如同时有 `.dng` 和 `.raf` 但主干相同，理论上不该发生但不排除用户手动整理产生的意外情况）之类的边界输入，具体判定/报错行为留给 Eng Design

## 未来考虑

* 更多 RAW 格式支持（CR2/CR3、NEF、ARW 等），等有对应器材需求时再加，Eng Design 阶段的格式判断逻辑需要为此留好扩展口子
* RAW 专属调整参数（highlight recovery、色差校正等），等真机使用中发现 M1 现有的高光/暗光/白平衡偏移不够用再考虑
* 导出进度反馈机制，如果 Eng Design 阶段的性能实测显示确有必要
