# PicZTream (PZT) 项目路线图（Roadmap）

## 项目背景

PicZTream 起源于一个具体且反复出现的痛点：每次旅行或拍摄归来，面对几百上千张照片，真正消耗时间的不是精修那一小部分值得投入的照片，而是"筛选"这个高频动作本身。现有工具无论是系统相册还是 Lightroom、Capture One，本质上都是为鼠标交互和缩略图网格设计的，切图延迟、UI 响应链路长，导致筛选阶段成为整个摄影工作流里最大的隐性时间黑洞。与此同时，RAW 格式虽然在精修阶段价值明确，但对于绝大多数看一眼就该被淘汰或直接发朋友圈的照片而言，RAW 的存储成本和处理复杂度是纯粹的负担，这也是用户长期想用 RAW 却因为麻烦而放弃的核心原因。

PicZTream 的解决思路是双层流水线：culling 阶段只读取相机 ISP 已经渲染好的内嵌 JPEG 或原生 JPEG 文件，绕过 RAW 解码，追求终端内全键盘、接近零延迟的秒切浏览体验；processing 阶段仅在用户明确标记某张照片需要精修时，才按需触发 LibRaw 解码与色彩处理。这种按需升级复杂度的设计，使得用户面对的默认体验永远是轻量的，只有少数真正值得投入的照片才会进入重量级流程。这个思路直接回应了 RAW 想用但太麻烦这个真实痛点，也是整个项目区别于市面工具的核心差异点之一，另一个差异点是终端全键盘工作流带来的心流不中断体验，这是任何 GUI 工具难以复制的。

项目的长期目标不止于本地工具。选片、打标签、分组这些动作在人工阶段积累的每一次判断，本质上是在为未来的自动化打偏好数据的地基。最终形态设想是：用户在旅行途中把照片通过即时通讯工具发送给一个中转账号，家中电脑上的 agent 自动完成初筛、打标签、生成配文候选，再把结果推送回用户做最终确认甚至全自动发布。但这个全自动愿景必须建立在一个前提上，就是本地核心引擎的选片逻辑、数据结构、色彩处理流水线已经被人工使用验证过、调校过，否则自动化只是在一个不稳固的地基上搭建黑盒。

## 开发环境与技术基调

* **硬件与终端：** Mac M 系列芯片 / Ghostty 终端（原生支持 Kitty 图像协议，可在 Tmux 窗格内渲染 100% 物理分辨率大图）/ Fish 4.0 / LazyVim
* **构建链：** CMake + Ninja
* **语言规范：** 现代 C++（C++20 及以上）
* **AI 协作原则：** 通过 Git Hook 与 Stow 锁死 Claude Code 与 agent 工具链的全局大原则——客观、严格、简洁、逻辑导向、追求极致低延迟与现代 C++ 规范

项目的技术基调是客观、严格、简洁、逻辑导向，追求现代 C++ 规范下的极致低延迟。核心色彩处理与并发调度完全依赖 C++（`std::jthread` 并发调度、ARM NEON SIMD 像素级优化）而非依赖外部脚本语言或运行时开销较大的框架，LLM 在这个项目中的角色被严格限定在算法推导和生成结构化配方（如 `recipe.json`）层面，不参与任何需要保证确定性和实时性能的核心执行路径。

## 双层流水线设计

1. **Culling 阶段（秒切）：** 指令级偏移解析 RAW 文件末尾相机 ISP 渲染好的内嵌高质量 JPEG，直接投喂给 Ghostty 渲染，绕过 RAW 解码，实现零延迟全键盘选片。
2. **Processing 阶段（P 图）：** 引入 LibRaw 抹平徕卡 Bayer（DNG）与富士 X-Trans（RAF）阵列去马赛克的计算代差，吐出统一的线性 `uint16_t` RGB 矩阵供上游色彩流水线消费。

## 算力分工

LLM（Claude / agent）仅负责高维算法推导及生成色彩配方 `recipe.json`；核心 3D LUT 色彩映射与白平衡偏移由 C++ 并发多线程（`std::jthread`）与 ARM NEON SIMD 直接处理像素二进制运算。

## 架构分层原则

架构层面从项目一开始就明确分层：核心业务逻辑（数据库读写、图片元数据管理、预取缓存、色彩处理流水线）被封装为独立的 `core` 库，不携带任何终端交互或渲染相关的依赖；终端全键盘交互界面作为 `cli` 前端调用这个 `core` 库。这个分层不是过度设计，而是为了让未来的 AI 辅助功能和 agent 项目能够直接复用同一套已经被验证过的核心引擎，只需要新增可插拔的异步旁路或 headless 编排层，而不需要重写任何底层算法。

## 里程碑总览

* **Milestone 0，MVP，选图管理核心。** 目标是验证全键盘、项目化、带上限标签分组这套交互模型是否成立，纯 JPEG 输入，零 AI 依赖，零网络依赖，产出一个可以立刻投入日常使用替代打开相册划拉这个动作的工具。这是唯一一个后续所有阶段都直接依赖其数据结构和交互范式的基础阶段。
* **Milestone 1，风格化，本地色彩预设与同步预览。** 在 MVP 稳定后引入 `recipe.json` 描述的色彩配方系统，支持键盘快捷切换风格并生成降采样同步预览，全分辨率处理走异步队列。这一阶段验证的是色彩处理流水线本身，先在纯 JPEG 场景下把参数和渲染逻辑跑通，避免一开始就绑定 RAW 解码的复杂度。
* **Milestone 2，RAW 支持，LibRaw 接入。** 在色彩处理逻辑已经稳定之后接入 LibRaw，正式打通双层流水线。此阶段严格遵循按需触发原则，只有用户标记为精修的图片才会真正触发 RAW 解码，绝大多数图片的浏览体验不受影响。
* **Milestone 3，AI 辅助功能，可插拔异步能力层。** 在核心数据结构和色彩流水线都已稳定之后，引入一组独立的 AI 能力，包括自动粗修（曝光校正、白平衡校正、水平化、简单裁切建议）、审美评分、自动打标签、以及近似重复图片检测去重。这些能力的共同前提是必须可插拔且异步：用户可以完全关闭其中任意一项而不影响 culling 主流程的零延迟体验，所有 AI 推理都在后台队列中运行，结果以建议的形式写回数据库供用户在浏览时参考，而不是直接替代用户的判断。此阶段与 Milestone 4 在时间线上相互独立，两者都只依赖 Milestone 0 到 2 沉淀的 `core/api`，彼此之间没有先后依赖关系，可以并行推进。
* **Milestone 4，Agent 半自动，即时通讯接入。** 复用前几个阶段沉淀的 `core` 库能力，新增一个 headless 编排层接收即时通讯工具发来的照片，自动完成初筛、打标签、生成候选文案，结果推送回用户做最终确认，人工始终保留最后拍板权。此阶段可以选择性地调用 Milestone 3 产出的 AI 能力作为其初筛逻辑的一部分，但即便 Milestone 3 尚未完成，也可以先用简单规则跑通编排链路，两者互为增强而非互为阻塞。
* **Milestone 5，全托管自动化，可选阶段。** 仅在 Milestone 4 的自动推荐质量经过实际使用验证足够可靠之后才考虑启动，目标是让 agent 完成选图到发布的完整闭环，人工只处理例外情况。这一阶段没有固定时间表，是否启动完全取决于前一阶段的数据表现。

## 阶段间的依赖关系

Milestone 0 到 2 是严格线性依赖的主干，后一阶段的可行性建立在前一阶段产出的稳定接口和真实数据之上，跳过任何一个阶段直接往后推进都会导致上层建筑失去地基。从 Milestone 3 开始出现分支：Milestone 3 和 Milestone 4 都只依赖主干上 Milestone 2 结束时的 `core/api`，彼此之间不构成依赖关系，可以根据实际精力选择先做哪一个，甚至交替推进。Milestone 5 则同时依赖 Milestone 3 的 AI 能力成熟度和 Milestone 4 的编排链路验证结果，是唯一一个需要两条分支都趋于稳定之后才具备启动条件的阶段。
