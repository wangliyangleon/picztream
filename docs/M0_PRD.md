# PicZTream (PZT) Milestone 0 产品需求文档（MVP）

## 背景与问题陈述

现有图片管理工具在大批量选片场景下存在结构性延迟，GUI 缩略图加载、鼠标交互路径长、评分与标签系统笨重，这些都在旅行归来面对几千张照片时被放大成难以忽视的时间成本。真正消耗时间的是筛选这个高频微操作本身，而不是精修，市面工具几乎没有一个是围绕这个洞察设计的。

## MVP 目标

验证全键盘、terminal 内 100% 物理分辨率图片浏览的可行性与延迟表现，验证项目加带上限标签的分组模型是否符合真实使用习惯，产出一个可以日常自用、能够替代打开相册应用划拉这个动作的工具。

## 非目标

* 不做 RAW 解码（Milestone 2 范畴）
* 不做任何色彩处理或风格预设应用（Milestone 1 范畴）
* 不做 AI 审美评分、AI 自动选图、AI 文案生成（Milestone 3 及之后范畴）
* 不做多设备同步或云端功能
* 不做图形界面，仅支持终端

## 目标用户

仅一个用户，即项目开发者本人，不考虑多用户、权限管理或协作场景，所有设计决策都以单人高频使用为唯一优化目标。

## 核心用户故事

* 用户可以将一批照片，例如一次旅行的整个文件夹，导入为一个项目
* 用户可以用键盘在项目内的图片间快速前后移动浏览，且切图没有可感知的延迟
* 用户可以对当前图片打上一个或多个标签，例如精选、朋友圈、废片
* 某些标签（例如朋友圈）有数量上限，当用户尝试超额添加时，系统会提示并允许选择替换已有的某一张，而不是简单拒绝操作或直接崩溃
* 用户可以按标签筛选查看图片子集，例如只浏览朋友圈标签下的图片
* 用户退出程序后再次打开同一个项目，之前的所有标签状态都完整保留
* 用户可以将某个标签下的图片导出到指定文件夹，方便后续手动发布

## 功能需求

### 项目管理

* `pzt new <project_name> [folder_path]`：扫描目标文件夹内的 JPEG 文件并创建新项目，元数据写入 SQLite；`folder_path` 可省略，省略时默认使用当前目录作为项目根路径；目标目录内一张 JPEG 都找不到时报错退出，不创建项目
* `pzt rescan <project_name> [--no-prune]`：项目建好之后，如果又往项目文件夹里新增了照片（扫描只在 `pzt new` 时发生一次，之后放进去的文件不会自动被发现），用这个命令补录磁盘上有、但项目里还没有的文件；默认同时会清除 `images` 表里有、但磁盘上已经找不到对应文件的记录，连带清掉这些图片打过的标签（级联删除）——increment 6.4 实际使用中发现"只增不减"会导致删掉的照片一直留在项目里挥之不去，比"清理时机不对"更让人困惑，因此改成默认清理；如果清楚自己是在对一个可能暂时不完整的存储位置（比如外置硬盘没挂载）跑 `rescan`，加 `--no-prune` 跳过清理，避免把还在的照片误判成已删除、丢失其标签
* `pzt open [project_name]`：`project_name` 可省略；带名字时按名字直接打开该项目；省略时根据当前所在目录匹配已注册项目的根路径并打开。两种情况下，如果找不到对应项目都报错退出，并提示用 `pzt list` 查看可用项目及其路径
* `pzt list`：列出所有已创建的项目及其基本统计信息，包括根路径、图片总数和各标签下的图片数量；已归档的项目排在列表最后并标注归档状态
* `pzt archive <project_name>`：将项目标记为已归档，不影响任何数据，只改变 `list` 里的展示分组
* `pzt delete <project_name>`：清除该项目在数据库中的全部元数据（项目、图片记录、标签、标签关联），不删除磁盘上的原始图片文件

### 浏览与选片（核心交互）

快捷键设计如下：

* `h` / `l`：上一张 / 下一张
* `j` / `k`：跳转到下一个 / 上一个尚未打标签的图片，用于快速定位待处理项
* `space`：打开标签快速菜单，数字键对应预设标签(按创建顺序固定,不是字母序;"废片"这个内置系统标签固定占 0 号位,不参与其它标签的动态编号)。`space` + 数字：给当前图片打上该标签,超出数量上限时状态栏提示已有条目并要求输入序号替换、或按 Esc 取消。`space -` + 数字：只从当前图片上摘掉该标签,不影响标签本身或其它图片。`space c`：新建一个标签。`space d` + 数字：删除该标签本身(连带清掉所有图片与它的关联,项目级操作,不只影响当前图片)；系统标签(目前只有"废片")不能删除,不会出现在这个删除列表里。不再保留独立的建/删标签命令,全部通过这个菜单完成
* `x`：标记为废片,这个内置系统标签的专用快捷键(等价于 `space` + 0,但更快,不需要先打开菜单)
* `g` + 数字：切换到指定标签的筛选视图，仅浏览该标签下的图片；数字键映射跟 `space` 菜单共用同一套编号（0 = 废片，1-9 按创建顺序对应用户标签），`g` + `0` 专门筛选废片
* `g` + `g`：清除筛选，回到完整项目视图
* `g` + `e` + 数字：导出指定标签（编号规则同上）下的图片到手动输入的目标文件夹，等价于不带 `--link` 的 `pzt export`（复制模式）；`g` + `e` + `e`：仅在当前处于 `g` + 数字筛选视图时可用，导出当前筛选标签，省去再选一次编号（规划中，见 `docs/M0_Eng_Design.md` increment 6.6）
* `q`：退出并保存状态

所有快捷键映射后续应支持在配置文件中自定义。图片通过 Kitty 图像协议直接渲染进 Ghostty 终端窗格，目标是达到物理原图分辨率且切图延迟不可感知。预取策略为当前索引前后各 N 张图片的内嵌 JPEG 提前解码进内存环形缓冲区，N 值可配置，默认值待性能测试后确定。

### 标签系统

用户可以自定义标签，每个标签可以选配一个数量上限，以及是否关心图片在标签内的排列顺序，例如朋友圈九宫格发布就需要关心顺序。标签与图片之间是多对多关系。当添加操作会导致超出数量上限时，交互方式是在状态栏提示当前已满的数量并要求用户输入序号来替换某一张已有图片或按 Esc 取消，全程不使用会打断心流的弹窗。标签的新建、删除，以及从单张图片上摘掉某个标签，都通过 `pzt open` 界面里的 `space` 菜单完成（见上一节快捷键设计），不提供独立的命令行子命令。"废片"是唯一的内置系统标签，`pzt open` 时确保存在（不存在就创建），固定占用编号 0，不能通过 `space d` 删除。

### 导出

通过 `pzt export <project_name> <tag_name> <output_folder>` 将指定标签下的图片按顺序（如果该标签关心顺序则按顺序编号）复制或软链到指定的输出文件夹。

## 非功能需求

* **延迟：** 单张图片从按键触发到渲染完成，目标控制在一百毫秒以内，前提是本地 SSD 读取且预取命中
* **数据持久性：** 所有标签操作需要实时写入 SQLite，或者至少保证程序意外退出时不会丢失当天已完成的选片结果
* **平台支持：** 本阶段仅支持 macOS M 系列芯片配合 Ghostty 终端与 Kitty 图像协议，不考虑其他终端的兼容性；同时支持在 Tmux 窗格内运行与直接在独立 Ghostty 窗口内运行（不经过 Tmux），由程序在启动时探测运行环境自动选择渲染路径

## 技术方案概要

开发语言为现代 C++，二十及以上标准，构建链为 CMake 配合 Ninja。图片输入直接读取内嵌 JPEG，具体来说是从 Exif 中的缩略图或预览图段读取，对于纯 JPEG 文件则直接读取原文件本身。渲染层使用 Kitty 图像协议直接输出到终端，同时支持 Tmux 窗格内运行（转义序列经 Tmux passthrough 包装）与独立 Ghostty 窗口内运行（不经过 Tmux，转义序列直接发送），具体路径由启动时的运行环境探测决定。存储层使用 SQLite，具体表结构留待下一步单独设计，预计核心表包括：

* 项目表（projects）
* 图片表（images）
* 标签表（tags）
* 图片与标签的关联表（image_tags）

架构上严格分层为不携带任何终端交互依赖的 `core` 库和调用该库的 `cli` 前端，这个分层从第一天起就是为了给 Milestone 3 和 Milestone 4 的接入预留稳定的 `core/api` 接口层。

## 验收标准

* [x] 可以从一个包含五百张以上 JPEG 的文件夹成功创建项目(`spikes/acceptance_500` 用 550 张合成 JPEG 验证过)
* [ ] 浏览五百张图片的全过程主观感受无可感知卡顿，并配合简单的延迟日志加以验证(延迟日志部分已验证:`spikes/acceptance_500` 跑 550 张合成图,avg 34ms/p95 37ms/无随会话推进变慢的趋势;**主观感受部分需要在真实终端里用真实照片过一遍**,合成图解码速度不具代表性,这条待你亲自确认)
* [x] 可以创建至少三个不同配置的标签，其中至少一个设置了数量上限，并且能够正确处理超额添加的场景(手动验证过:三个不同配置的标签,一个设置了 cap=2,超额添加时正确弹出替换提示并完成替换)
* [x] 关闭程序后重新打开同一项目，标签状态完整保留，不发生任何丢失(手动验证过:多次独立进程重开项目,标签关联数量保持不变)
* [x] 可以成功将某个标签下的全部图片导出到目标文件夹(手动验证过)
* [x] 可以将项目标记为归档，`list` 中归档项目排在最后并标注归档状态；可以删除项目的数据库元数据且不影响磁盘上的原始图片文件(手动验证过:归档后 `list` 正确显示 `[已归档]`,删除后元数据消失但磁盘文件保留)
* [x] 核心逻辑（标签的增删、数量上限校验、项目扫描）具备基本的单元测试覆盖(`ctest`:71 个用例、642 个断言,全部通过)

## 风险与待确认问题

* Kitty 图像协议在 Tmux 窗格内的实际渲染延迟目前尚未经过实测，需要在早期做一次独立的验证性测试，如果实际延迟不符合预期，需要评估是否要提供一个降级到较低分辨率预览图的兜底方案
* 环形缓冲区的大小与内存占用之间存在权衡，大批量高分辨率 JPEG 如果全部预取可能带来较大的内存压力，需要设定一个经过测试的合理默认值并允许用户按需配置
* SQLite 的写入频率，也就是每次打标签操作都立即写盘还是采用批量或定时写入策略，对交互延迟的实际影响也需要通过实测来确定最终方案
