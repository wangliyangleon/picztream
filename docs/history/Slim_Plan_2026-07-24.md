## 1. 概览

### 方法

先建立测试基线（第 0 步），再识别正确性关键路径（第 1 步），然后按模块通读源码，用重复模式、无引用导出、`TODO`/`FIXME`/`legacy` 标记、函数长度、`git` 现状等信号定位候选。每条结论都附代码位置（文件 + 行号）。

### 测试基线（Step 0）

| 套件 | 命令 | 结果 |
| --- | --- | --- |
| C++ core | `cd build && ctest`（Debug，含 ASan/UBSan） | `core_tests` 通过，**297** 个 doctest case |
| C++ cli | 同上 | `cli_tests` 通过，**38** 个 doctest case |
| Python agent | `cd agent && .venv/bin/python -m pytest -q` | **367 passed, 1 warning** |

说明：

- ctest 只驱动两个测试二进制（`core_tests`、`cli_tests`），二者均 100% 通过。
- agent 的 1 条 warning 来自 `tests/transport/test_telegram.py` 里 `TelegramTransport._poll_loop` 协程未 await 的既有噪声（"Task was destroyed but it is pending"），与本计划无关，属基线的一部分。
- 后续任务验收里"测试全绿"= **匹配此基线**（297 / 38 / 367），不是"比基线更多"。

**覆盖盲区（影响可重构范围）**：`cli/tests/` 只覆盖 `i18n` / `text` / `kitty` 三个纯函数模块。**命令层（`cli/commands/*.cpp`，含 `cmd_open` 交互主循环）没有单元测试覆盖。** 按本计划的可读性重构闸门，未覆盖代码只能"先补 characterization 测试"或"不动"，不做盲改。

### 正确性关键路径（Step 1）

以下路径的重复/防御性检查**推定为有意为之**，不进删除队列，最多标"needs human confirmation"：

| 路径 | 位置 | 说明 |
| --- | --- | --- |
| 不碰用户原始文件 | `core/project`、`core/export` | 一切状态入 PZT 自己的库，原图只读；重复的路径/存在性校验是设计不变量。 |

这是一个**维护度很高、已相当精简**的代码库：几乎没有 `TODO`/`FIXME`（生产代码里仅 1 处 `TODO`、3 处 `legacy` 且都是有意保留并带注释说明），单实现抽象要么有真实多态需求（三个 AI provider）、要么干净委托（`request_json` 单图重载直接转调多图重载），最近 `W2026-07-21` eval 解耦后也没留下 `overall_score`/`passes_gate` 的死引用（仅剩历史说明注释）。因此**低风险精简空间很小**。

## 2. 发现列表

### 冗余代码

#### F-1 · `adjustment_parser.py` 简单分类器样板重复

- **位置**：`agent/compose/adjustment_parser.py`
  - `classify_style_describe`（305-316）
  - `classify_style_gate_reply`（339-350）
  - `classify_running_message`（424-435）
  - `classify_cancel_confirmation`（456-467）
  - 部分相关：`classify_collecting_message`（264-276）、`classify_dedup_followup`（389-404）
- **描述与证据**：上述至少 4 个函数体是逐字同构的：
  ```python
  decision = request_json(user_prompt=..., schema_instruction=_XXX, provider=meta_provider, http_post=http_post)
  action = decision.get("action")
  if action not in (...): raise AdjustmentError("unknown_action", f"unrecognized ... action {action!r}")
  return XxxReply(action=action)
  ```
  差异只在三处：schema 常量、允许的 action 元组、返回的 `*Reply` 类名。这是"近似相同逻辑可合并"，不是有意的防御性重复（分类器不在正确性关键路径的删除禁区内，`classify_cancel_confirmation` 的**安全语义在 schema 文案里**，不在这段样板里，抽取不影响它）。
- **建议修复**：在文件内加一个**私有**辅助函数承载"调用 + 取 action + 校验 + 报错"这段确定性管线，例如：
  ```python
  def _classify_action(user_prompt, schema, allowed, http_post, meta_provider, label):
      decision = request_json(user_prompt=user_prompt, schema_instruction=schema,
                              provider=meta_provider, http_post=http_post)
      action = decision.get("action")
      if action not in allowed:
          raise AdjustmentError("unknown_action", f"unrecognized {label} action {action!r}")
      return action
  ```
  4 个"单字段 Reply"的分类器改为 2-3 行调用；schema 常量与 `*Reply` dataclass **保持显式不动**。`classify_collecting_message` / `classify_dedup_followup` 各有 user_prompt 构造 / `narrow` 特例，可选择只复用校验核心、或不纳入本次（见验收）。
- **风险**：低。`tests/compose/test_adjustment_parser.py`（315 行）直接测公开函数，抽取属行为保持重构，测试无需改断言。
- **判断说明**：这是 DRY 合并同构代码，不是投机性可扩展抽象，符合项目"不做超范围抽象"约束（辅助函数私有、同文件、无新扩展点）。属**可选优化**，非必须；执行者须确认 pytest 结果与基线逐字一致。

### 可读性

#### F-2 · `cmd_open` 超长函数（~950 行）

- **位置**：`cli/commands/browse.cpp:526-1478`（`int cmd_open(...)`）。
- **描述与证据**：单函数近千行，承载全键盘浏览主循环（导航 / 渲染 / prefetch / 标签菜单 / `/filter` 控制台 / 废片 / recipe 切换 / 导出 / 延迟统计）。函数内注释质量很高，但体量与嵌套深度客观上属可读性负担。
- **建议修复**：**暂不重构。** 该函数满足两条"不动"条件：(1) 是零延迟交互热路径中的输入分发循环；(2) `cli` 层无任何针对命令处理器的测试覆盖（见基线覆盖盲区）。按 skill 规则，行为等价重写恰恰是 error-handling 顺序 / 边界易埋 bug 的最高风险类别，**未覆盖代码不做盲改**。
- **风险**：高。
- **Flags**：`insufficient test coverage`、`needs human confirmation`。若确要处理，正确的前置任务是"为 `cmd_open` 的按键分发补 characterization 测试"，那需要先设计一层可测的输入/渲染缝（属超出本 slim-plan 范围的架构改动，应先立 PRD）。

#### F-3 · `resolve_scope` vs `resolve_console_scope`

- **位置**：`cli/commands/commands.cpp:110`（`resolve_scope`，headless 用，错误走 `error_code`/`error_msg`）与 `cli/commands/browse.cpp:205`（`resolve_console_scope`，交互用，错误走 i18n 人读文案）。

- **风险**：中（跨 `core`/`cli` 边界，涉及错误呈现语义）。
