# PicZTream (PZT) Milestone 3 产品需求文档（近似重复检测）

## 背景与问题陈述

`docs/Roadmap.md` 定义的 M3 四个能力之一是"近似重复检测"，`docs/M3_PRD.md`（选片辅助评估）里明确把它排除在外、单独立项——两者虽然都服务于"选片"这个大目标，但技术形状完全不同：选片辅助评估是"对单张图片发一次云端 AI 请求"，近似重复检测是"本地算法比对一批图片"，不涉及云端调用。

连拍、包围曝光、反复试拍同一个构图，都会在一个项目里留下大量内容几乎相同的照片——这些不是"值不值得留"的问题（技术上可能都合格），而是"这些其实是同一次按快门产生的好几张，只需要留一张"的问题。人工从几百上千张照片里肉眼挑出这些重复组，费时费眼睛；这个能力要解决的就是这个。

唯一的触发入口是 `pzt open` 里 `:` 控制台的命令——`/dedup *` / `/dedup #标签名`（`/` 命令惯例见 `docs/M3_PRD.md`"触发入口"一节）。不做成独立的顶层 CLI 命令：既然已经是"打开 `pzt open` 之后按需触发"的操作，没必要再维护第二个入口。

## M3 目标（本增量范围）

* `pzt open` 的 `:` 控制台新增 `/dedup *` / `/dedup #标签名` 命令——`*` 是整个项目，`#标签名` 缩小范围到带这个标签的照片；不带额外指引参数（dedup 不是问 AI，没有"额外指引"这回事）
* 这次是**阻塞式**的：命令触发之后 `pzt open` 会冻结几秒到几十秒直到跑完（dedup 是纯本地操作，这次不为它单独搭一套异步基础设施，先用最简单的方式跑通；要不要做成不阻塞浏览的异步版本，留到以后有真实使用体验再评估）
* 判断近似重复分两步：
  1. 先按拍摄时间（`captured_at`）把时间上紧挨着的照片分成候选簇——连拍/包围曝光产生的重复几乎总是在时间上挨得很近，这一步避免对整个范围做 O(n²) 全量比对，也避免"两张时间上毫不相关的照片碰巧感知哈希接近"这种误判
  2. 候选簇内部用感知哈希（具体算法留给 Eng Design）比较照片的视觉相似度，相似度超过阈值的归进同一个"重复组"
* 每个重复组里，自动选出一张"保留"，其余的自动打上一个系统标签（比如"duplicate"，复用现有 tagging 系统，跟"废片"标签同样的落地方式）标记为重复，不删除任何文件、不改动数据库之外的任何东西
* 选保留哪一张的规则，**按组独立判断，不是整个范围一刀切**——范围里有的组全员评估过、有的组缺评估，两种组各自按自己的情况选，不会因为范围内某个不相关的组缺评估就连累其它组也退化：
  * 这一组内**所有**成员都已经跑过选片辅助评估（见 `docs/M3_PRD.md`）时，按综合分数（`overall_score()`）最高的保留——"选质量最高的"这个判断本身依赖评估结果，只有这一组全员都有分数时才有资格这么选
  * 这一组里只要有一张没评估过（不管是这一组全没评估、还是评估了一部分），这一组就不比较分数，改成按 `captured_at` **最新**的保留——这不是在假装"最新拍的就是质量最好的"，只是在没有可靠质量信号时选一个确定性强、至少讲得通的默认值
  * 保留规则用来自动选一个"默认保留项"，不代表用户不能改——用户随时可以手动摘掉"duplicate"标签、换一张保留，这只是省去从头开始比较的工作，不是不可更改的最终决定
* **不代为触发评估**：触发之后先检查一遍范围内的照片是不是都评估过了，没有就弹出一个确认提示（"N 张照片还没评估，是否继续？(y/N)"），按 `y`/`Y` 才继续（这次的保留判断大概率会有一部分组退化成按拍摄时间选），按其它任何键（包括 Esc）都当取消；不会自动帮用户发起评估请求——评估永远是用户自己主动决定要不要做的事（按 `:` 单张，或者用 `/ai_eval` 批量，见 `docs/M3_PRD.md`）
* 跑完之后用状态栏展示结果摘要（分组数、标记数），跟其它顶层键的反馈方式一致
* **重新运行**：每次运行前先把**这一次范围内**所有照片的 duplicate 标签清空，再重新跑一遍完整算法——不做增量对比，每次都是干净重来。范围是关键词：`/dedup *` 清空整个项目的旧标记，`/dedup #标签名` 只清空带这个标签的照片身上的旧标记，不动范围外的照片（哪怕它们是之前某次全项目扫描留下的标记）

## 非目标

* **不做独立的顶层 CLI 命令**——唯一入口是 `pzt open` 里的 `/dedup`，不维护 `pzt dedup <项目名>` 这样一个脱离 `pzt open` 也能跑的命令
* **不做自动触发**——这次只支持手动触发 `/dedup`，不会在 `pzt new`/`pzt rescan`/`pzt open` 空闲时自动跑重复检测，触发条件、频率控制这些留到以后（如果真的需要）单独设计
* **不做成异步**——阻塞 `pzt open` 几秒到几十秒是这次刻意接受的代价，不为了不阻塞浏览而新搭一套异步 worker
* **不做云端/AI 判断**——整个检测流程是本地确定性算法（时间聚类 + 感知哈希），不调用 Claude/Gemini，不产生任何 API 调用成本
* **不代为触发评估**——`/dedup` 只检查、提醒，不会自动帮用户发起选片辅助评估请求；评估和去重是两件各自独立、用户各自主动触发的事，不做成一条自动链路
* **不做自动删除文件或数据库记录**——"重复"只是打标签，不删除任何东西，即便判断错了也可以随时通过摘标签恢复，不会造成数据丢失
* **不做跨项目的重复检测**——只在单个项目内部比对，不同项目之间的照片不互相比较
* **保留规则不追求完美**——评估分数排序、拍摄时间兜底都是启发式规则，不保证每次都选出"客观上最好"的那张，这次先跑通"自动分组 + 自动打标签 + 用户可覆盖"这条链路，规则本身留到真实使用之后再调优

## 核心用户故事

* 用户正在 `pzt open` 里浏览，觉得手头这批照片（整个项目，或者手动打了个临时标签的一部分）好像有重复，按 `:` 输入 `/dedup *` 或者 `/dedup #标签名`，等个几秒到几十秒看结果
* 触发时如果范围内大部分照片还没评估过，会先看到"87 张照片还没评估，是否继续？(y/N)"这样的提示——按 `y` 继续（这次的保留判断大概率会退化成按拍摄时间），按别的键取消，先去用 `/ai_eval` 批量评估完再回来跑
* 用户在 `pzt open` 里用 `g` 筛选出打了"duplicate"标签的照片，一组一组看过去，确认自动选的保留项没问题就跳过，觉得选错了就手动调整标签
* 用户对某个重复组里"哪张该保留"的自动判断不认同，直接在 `pzt open` 里手动摘掉那张的 duplicate 标签、给该保留的那张打上（或者反过来），这个能力不会跟现有的手动打标签流程冲突或者覆盖用户的判断

## 功能需求

### 触发入口

`pzt open` 里 `:` 控制台命令 `/dedup *` / `/dedup #标签名`（`/` 命令惯例见 `docs/M3_PRD.md`"触发入口"一节，不带额外指引参数）。触发之后：

1. 先算出这次范围内有多少张照片还没评估过
2. 有的话，弹出一个单键确认提示（复用现成的单键读取原语，不是新的输入机制）——按 `y`/`Y` 继续，按其它任何键（含 Esc）取消，不跑
3. 确认继续（或者范围内本来就全部评估过，不需要问）之后，才真正开始比对——这一步阻塞 `pzt open`，跑完用状态栏展示结果（分组数、标记数），跟其它顶层键的反馈方式一致

### 分组与标记

见"M3 目标"一节的两步流程和保留规则。标记用的系统标签第一次运行时自动创建（幂等，照抄 `ensure_reject_tag` 的先例），不需要用户提前手动建好。

**重新运行**：每次运行都先把这一次范围内所有照片身上的 duplicate 标签清空，再重新跑一遍完整的分组+标记流程——不尝试"这次和上次比多了/少了什么"这种增量对比，也不区分"这个标签是上次自动打的还是用户手动加的"。用户手动调整过的 duplicate 标记，只要不主动在同样的范围重新触发一次 `/dedup`，就不会被改动；一旦重新触发，就是完全按这一次的算法结果重新来过，之前的手动调整不会被保留。

## 非功能需求

* **性能**：单个项目几百到几千张照片的量级下，扫描应该在几十秒内完成；时间聚类这一步的存在就是为了避免真正昂贵的感知哈希比较发生在明显不相关的照片对上
* **确定性**：同样一批照片、同样的参数，多次运行结果应该一致（纯本地算法，没有 AI 调用的不确定性）

## 技术方案概要

具体的感知哈希算法选型（dHash/pHash）、时间聚类窗口、相似度阈值、"重复组"数据怎么落库、保留规则的精确实现，留给 Eng Design 阶段确定，这份 PRD 只锁定产品行为。

## 验收标准

* [ ] `pzt open` 里按 `:` 输入 `/dedup *`，能对整个项目跑一遍分组
* [ ] `/dedup #标签名` 能正确把范围限定在带这个标签的照片，不影响项目里其它照片
* [ ] 对一批人工构造的连拍测试照片（同一场景、时间上连续、内容几乎相同），能正确把它们分进同一个重复组
* [ ] 对时间上连续但内容明显不同的照片（比如同一分钟内拍的完全不同的两个场景），不会被误判为重复组
* [ ] 一个重复组内所有成员都评估过时，保留的是 `overall_score()` 最高的那张
* [ ] 一个重复组内只要有一张没评估过，保留的是 `captured_at` 最新的那张，不看分数
* [ ] 范围内有未评估图片时能看到 y/N 确认提示，按 `y`/`Y` 才会真正开始跑，按其它键（含 Esc）能正确取消、不产生任何标签改动
* [ ] 运行期间不会发起任何 AI 评估请求（真机验证：跑的时候没有网络请求，也没有排队的评估任务）
* [ ] 跑的时候 `pzt open` 确实会冻结（不是异步），跑完之后正常恢复响应
* [ ] 完成后能在状态栏看到清楚的摘要（重复组数、标记数）
* [ ] 不修改任何图片文件本身，只操作数据库里的标签
* [ ] 用户手动摘掉某张照片的 duplicate 标签之后，只要不重新触发同样范围的 `/dedup`，这个调整会一直保留
* [ ] 重新触发同样范围的 `/dedup`，之前所有手动调整都会被这一次的算法结果重新覆盖（这是预期行为，不是 bug）；重新触发只清空这次范围内的旧标记，不影响范围外的照片

## 风险与待确认问题

* **相似度阈值怎么定**：感知哈希的汉明距离阈值定太松会把不相关的照片误判成重复，定太紧会漏掉真正的重复——这次没有真实数据集调参，初版阈值靠经验值起步，真实使用之后再调
* **时间聚类窗口怎么定**：多长时间内的照片算"候选相邻"？连拍可能几秒内几十张，包围曝光可能间隔稍长，具体窗口值留给 Eng Design，可能需要做成可调参数而不是硬编码常量
* **重复检测的计算量**：感知哈希本身很快，但如果一个候选簇异常大（比如一次导入了几百张连拍），簇内仍然是 O(k²) 比对——这个量级下会不会成为性能问题，需要 Eng Design 阶段估算或者实现后用真实数据验证
* **保留规则的兜底是否合理**：`captured_at` 最新只是一个"没有更好信号时讲得通"的默认值，不代表真的对应"更值得留"，真实使用之后可能需要重新评估（比如要不要换成本地清晰度检测这类不依赖 AI 评估也能衡量质量的信号）
* **阻塞期间的输入缓冲行为**：`pzt open` 冻结的这几秒到几十秒里用户按的键会发生什么（丢弃？缓冲到跑完之后突然触发？），这次没有专门设计，取决于终端/内核 tty 缓冲区的默认行为，实现阶段需要真机验证并且在文档里记录下来，不能是"没想过、跑起来才发现"的状态
* **要不要做成异步**：这次刻意先做阻塞版本，如果真实使用中发现"跑一次 dedup 要冻结 pzt open 半分钟"体验上无法接受，需要重新评估要不要仿照 `EvaluationWorker` 做一个异步版本——这会是一次不小的返工，这次先接受这个风险
