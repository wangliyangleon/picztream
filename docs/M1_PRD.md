# PicZTream (PZT) Milestone 1 产品需求文档（风格化）

## 背景与问题陈述

Milestone 0 验证了全键盘、项目化、带上限标签分组这套选片交互模型是成立的，核心数据结构（`core/api`）和终端渲染链路（Kitty 协议 + 预取缓存）已经稳定，可以日常自用。Milestone 1 要验证的是双层流水线设计里的另一半：色彩处理流水线本身能不能在终端全键盘工作流里做到跟选片一样的"零延迟切换"体验。这一步刻意只在纯 JPEG 场景下把参数和渲染逻辑跑通（`docs/Roadmap.md` 原话），不提前绑定 RAW 解码（LibRaw 接入是 Milestone 2 范畴）的复杂度。

## M1 目标

引入 `recipe`（色彩配方）系统：用户可以给图片应用一个预设风格，浏览时看到同步渲染的降采样预览，退出重开项目后应用状态完整保留；同时验证"全分辨率处理"在真机上是否需要异步队列（Phase 0 spike 已给出初步数据，见下文"技术方案概要"）。

## 非目标

* 不做 RAW 解码（Milestone 2 范畴）
* 不做直方图、曲线等专业调色 UI——PZT 的定位是终端全键盘快速选片工具，不是精修软件；`recipe` 是预设风格的整体切换/应用，不提供逐参数手动调节的界面
* 不做 `recipe` 的导入导出机制——M1 里 `recipe` 完全通过 `pzt open` 内的交互创建、存在数据库里，不存在独立的 `recipe.json` 文件形态；这是主动的范围裁剪，不是遗漏，见"未来考虑"
* 不做 LLM 在 pzt 运行时动态生成配方——呼应 `CLAUDE.md` "AI 使用边界"：LLM 负责算法推导/生成结构化配方，不参与运行时的核心执行路径；等导入机制真正落地后，"线下用 LLM 起草一份配方再导入"才有意义，这次不涉及
* 不修改磁盘上的原始 JPEG 文件——这是延续自 M0 的承诺（`archive`/`delete` 都不碰文件本身），M1 引入的色彩处理同样不例外，见下文"持久化与撤销模型"

## 核心用户故事

* 用户可以给一张图片应用一个预设风格，浏览时立刻看到应用后的效果（降采样预览，同步渲染，不可感知延迟）
* 用户可以随时把一张图片的风格换成另一个，或者取消风格、恢复成原图效果——这个过程不涉及任何文件层面的改动，原始 JPEG 从始至终没有被写过
* 用户可以在某个内置预设的基础上调整高光、暗光、白平衡偏移等细节，存成自己的一个版本，之后随时再选中应用
* 用户退出程序后再次打开同一个项目，之前应用的风格状态完整保留
* 用户删除一个自己创建的版本后，已经用过它的图片不受任何影响，继续保持原来的效果展示；只是这个版本以后不会再出现在可选列表里
* 用户导出某个标签下的图片时，应用了风格的图片按处理后的效果导出，没应用风格的图片继续按 M0 已有的行为导出原始文件

## 功能需求

### 持久化与撤销模型（这个里程碑最核心的架构决策）

`recipe` 分两层，参照富士相机"胶片模拟 + 自定义细节"的模型，不是简单的一张扁平列表：

* **第一层：胶片模拟预设（preset）**——内置、固定，随 pzt 发布，存在 SQLite 里（暂定 `presets` 表：id、name UNIQUE、is_system、色彩定义本身待 Eng Design 定）。每个预设有自己独特的整体色彩风格，预设本身就是"所有可调细节都是 0"的那个中性版本，不需要单独建一个"version 0"的子行。**预设不可删除、不可重命名**，M1 也不支持用户新建预设——新建预设意味着要么写新的基础色彩算法，要么导入别人做好的，后者已经推迟到"未来考虑"。
* **第二层：用户在某个预设基础上调整细节后保存的 version**——细节包括高光、暗光、白平衡红/蓝偏移等（具体参数集合待 Eng Design 阶段定），必须挂在某一个预设下面（暂定 `recipe_versions` 表：id、preset_id 外键、高光/暗光/白平衡等参数列、`deleted_at` 可空、created_at）。

一张图片应用某个风格 = 图片记录上有一个指向某个 `recipe_version` 的可空外键引用，同时只生效一个（不允许叠加多个，类比 Lightroom 一次只应用一个 Profile，保持模型简单）。

**固定的"Origin"选项**：代表"不应用任何风格"，固定占据编号 0（参照"废片"固定占 `space`/`g` 菜单 0 号位的先例）。它不是一张真实的数据行，纯粹是"清除当前图片的风格引用"这个动作在选择菜单里的可发现入口——因为没有细节可调，选它不需要像其它预设那样再进一层选 version。

**软删除，不是级联清除**：删除一个 version 不会影响正在用它的图片——这跟删标签不一样，标签只是分类元数据，删完顶多列表里少一项；但 `recipe` 直接决定图片观感，级联清空会让用户觉得"图片突然变了"，违反直觉。做法是给 `recipe_versions` 加 `deleted_at`：删除只是标记，把它从"应用/创建"的可选列表里隐藏，但已经引用它的图片渲染完全不受影响，数据没有被清空。`pzt recipe list` 里软删除的 version 依然显示、标注"已删除"并排在后面——直接复用 M0 `pzt list` 里"归档项目照样列出、标注归档、排在最后"这个已经验证过的模式。没有任何图片再引用的软删除 version，是否要在某个时机（比如 `pzt new`）顺手真正清理掉，还是干脆不清理，是可选的优化项，不是这次的硬性要求。预设（第一层）本身不受这条规则影响，预设从一开始就是不可删除的。

因为原始 JPEG 从来没被写过，风格相关的所有操作在任何时候都是安全的、可逆的——这也是选择这套非破坏性模型而不是"覆写原文件 + 存隐藏备份"方案的核心原因。

### 应用与预览

浏览界面（`pzt open`）新增顶层前缀键 `r`（recipe），结构对齐现有 `space`/`g` 菜单体系：

* `r` + 预设数字 + version 数字：应用（两层选择）。例：`r 2 0` = 应用 2 号预设的中性版本，`r 2 1` = 应用该预设下用户保存的第 1 个自定义版本。`r` + `0` 直接应用固定的 Origin，不需要第二层
* `r` + `r`：恢复原图（清除当前图片的风格引用），跟 `r` + `0` 效果相同，只是不经过菜单的两键快捷方式，对齐 `g` + `g` 清筛选的模式
* `r` + `c`：交互式创建新 version——选一个基础预设、调整细节旋钮、存成该预设下的新 version，对齐 `space c` 创建标签的模式
* 删除同样是两层选择（先选预设、再选该预设下要删的 version），预设本身不出现在删除列表里（跟系统标签不出现在 `space d` 删除列表是同一个道理），删除的效果是上面"软删除"，不是级联清图片引用

选中即应用，不需要额外确认弹窗，参照标签系统"`space` + 数字直接打标签"的交互哲学——风格切换和打标签是同等重量级的轻量操作。

预览是当前图片的降采样版本经过 `recipe` 处理后的实时渲染，同步完成，不引入额外的后台线程或缓存机制——这个结论由 Phase 0 spike（`spikes/color_lut_probe/results.md`）实测数据支撑，预览分辨率下即使最贵的 3D LUT 三线性插值单线程也只要 10-22ms，比 100ms 目标有 5-10 倍余量。

### 命令行

* `pzt recipe list [project_name]`：列出所有预设及其下属 version（含软删除的，标注状态）
* `pzt recipe rename <old> <new>`：给某个 version 改名（具体怎么在命令行寻址"某预设下的某个 version"，比如 `chrome:1` 这种写法，留给 Eng Design，见"风险与待确认问题"）
* `pzt recipe delete <preset> <version>`：软删除某个 version（寻址语法同上）

**没有** `pzt recipe create`：新建 version 只能通过 `pzt open` 里的 `r c` 交互流程完成，不提供纯命令行创建路径，跟标签系统现状一致（标签也没有独立于交互菜单之外的创建入口）。

### 导出

导出（`pzt export`）遇到应用了 `recipe` 的图片时，导出处理后的像素（"烘焙"进最终文件），不是原始文件的直接拷贝；没有应用 `recipe` 的图片继续保持 M0 已有的行为（复制或软链原始文件字节，不解码不重新编码）。这一步是全分辨率处理，走异步队列（Phase 0 spike 已经确认这一档必须异步：12MP 单线程处理就踩线 100ms，60MP 档要 725ms），不阻塞导出命令本身的其它部分。

## 非功能需求

* **预览延迟：** 同 M0，单张图片从按键触发到渲染完成目标控制在 100ms 以内，前提是本地 SSD 读取且预取命中；色彩处理这一步由 Phase 0 spike 验证过在预览分辨率下有充足余量
* **数据持久性：** `recipe` 定义与图片对它的引用都需要实时写入 SQLite，原始 JPEG 文件在任何情况下都不会被修改或覆写
* **导出正确性：** 应用了 `recipe` 的图片导出必须体现处理后的效果；未应用的图片导出必须是原始文件字节的精确拷贝（不能因为引入了色彩处理管线就意外重新编码所有导出图片）
* **平台支持：** 沿用 M0，仅支持 macOS M 系列芯片配合 Ghostty 终端与 Kitty 图像协议

## 技术方案概要

色彩处理管线基于 3D LUT 色彩映射（三线性插值）+ 简单的白平衡/曝光类逐像素变换，具体参数集合待 Eng Design 阶段设计 `recipe_versions` schema 时确定。并发调度沿用项目既有约定，统一用 `std::jthread`：全分辨率处理按行/按块切分到多个 `jthread` 上跑，不引入新的并发原语。**不引入 ARM NEON SIMD**——Phase 0 spike（`spikes/color_lut_probe/`）实测数据显示 `jthread` 多线程已经把 60MP 档全分辨率 LUT 应用压到 165ms（后台异步、不阻塞交互主循环），没有数据支撑现在就投入 SIMD 优化，符合 `CLAUDE.md` "不做过早优化"的工程契约；真机使用后如果后台队列吞吐仍是瓶颈，再回头用实测数据评估。

Phase 0 spike 同时确认了一个对预设/版本参数设计有直接影响的结论：3D LUT 网格大小（测过 17³ 和 33³）对性能几乎没有影响——网格选多大是纯粹的调色精度权衡，不需要顾虑性能，两者可以完全解耦决策。

架构上延续 `core`/`cli` 分层：预设/`version` 的 CRUD、应用/撤销、导出烘焙都是 `core` 业务逻辑，`cli` 只负责按键交互和终端渲染。

## 验收标准

* [ ] 可以在某个内置预设基础上创建一个自定义 version，应用到某张图片上，浏览时能看到预览效果
* [ ] 关闭程序后重新打开同一项目，图片应用的风格引用完整保留
* [ ] 软删除一个 version 后，已经引用它的图片渲染不受任何影响；该 version 不再出现在 `r` 菜单的可选列表里；`pzt recipe list` 仍然显示它并标注已删除
* [ ] 导出应用了 `recipe` 的图片时，导出文件体现处理后的效果；未应用 `recipe` 的图片导出保持原始文件字节不变
* [ ] 内置预设不能通过常规删除流程被误删或改名
* [ ] 核心逻辑（预设/`version` CRUD、应用/撤销、软删除、导出烘焙）具备基本单元测试覆盖
* [ ] 浏览时切换风格的预览延迟不可感知，配合延迟日志验证（复用 M0 已经建立的 `key-to-render` 汇总机制）

## 风险与待确认问题

* **`recipe_versions` 具体参数集合未定**：高光、暗光、白平衡红/蓝偏移这几个是确定要有的，还有没有别的维度（饱和度、颗粒感之类），需要在 Eng Design 阶段基于真实调色需求定下来
* **内置预设风格的具体数量、名称、色彩倾向未定**：结构已经定了（第一层预设 + 第二层用户版本），具体内容需要实际调色设计工作，留到实现阶段
* **`pzt recipe rename`/`delete` 的命令行寻址语法未定**：怎么在命令行里指代"某预设下的某个 version"（比如 `chrome:1` 这种写法），留给 Eng Design
* **软删除 version 的清理时机未定**：有没有必要在某个维护时机（比如 `pzt new`）顺手清掉没有任何图片引用的软删除 version，还是干脆不清理，标记为可选，不强制
* **软删除没有对应的"恢复"操作**：跟 `docs/M0_Eng_Design.md` 里 `archive` 没有 `unarchive` 是同一类悬而未决状态，如果以后发现这是常见需求再补
* **色彩管理（ICC）缺口维持 M0 的假设**：继续信任 CoreGraphics 默认 ColorSync 匹配，不在 M1 做显式 ICC 处理；这个简化在 M0 只影响解码显示，M1 引入真正的色彩处理管线后风险可能更大（如果 3D LUT 是针对某个假设的工作色彩空间设计的，跟源文件实际色彩空间不匹配可能放大偏色问题），先不处理，等真机使用中发现明显偏色问题再回头评估

## 未来考虑

* **`recipe` 的导入导出（`recipe.json` 交换格式）**：这次主动推迟，不算风险。等有真实的"分享风格给别人"或者"线下用 LLM 生成配方再导入"这类需求时，再设计具体的 schema 和 `pzt recipe import`/`export` 命令
