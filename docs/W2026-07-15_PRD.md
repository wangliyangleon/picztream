# PicZTream (PZT) 周开发目标（2026-07-15）

## 背景与定位

M0-M4 各里程碑的 PRD/Eng Design 覆盖的是"一整个里程碑"粒度的规划，颗粒度较粗、跨度较长。从本周开始，在里程碑规划之上叠加一层更细的**周开发目标**节奏：每周一份"开发目标文档"（本文档，关注需求和 deliverable，格式类似 PRD）+ 一份对应的"Eng Design 文档"（关注工程实现、模块划分与 tradeoff，规划阶段结束、进入实现前产出）。命名沿用现有 `M{n}_PRD.md`/`M{n}_Eng_Design.md` 的扁平放置约定，用日期代替里程碑号：`docs/W{YYYY-MM-DD}_PRD.md` / `docs/W{YYYY-MM-DD}_Eng_Design.md`，日期取该周开发目标定稿的那一天。

本周（2026-07-15 当周）确定的四个开发目标横跨三个既有里程碑的遗留缺口：M3/M4 的本地模型（AI 相关基础设施）、M1 遗留的 recipe/滤镜系统（几乎从 proof-of-concept 状态起步）、M4 的 agent Style 能力（未实现）、以及此前完全空白的部署与分发。四者分属不同子系统（C++ AI provider / C++ 色彩流水线 / Python agent / 打包脚本），耦合度低，可并行推进，具体见文末"建议执行顺序"。2026-07-17 追加目标五（agent 运行时重构）：前三个目标完成后真机使用暴露的体验问题，见该节。

## 目标一：本地模型支持（Provider::Local / Ollama）

**问题**：云端评估（Claude/Gemini）依赖用户自己的 API key 额度，目前连日常开发验证都会被限流卡住，需要一条不依赖外部配额的本地评估路径。

**范围（已拍板：只做 Tier 1，不做 Tier 0）**：
- 给 `core/ai/ai.h` 的 `Provider` 枚举新增 `Local`，接 Ollama 的 OpenAI 兼容 HTTP 接口（默认 `http://localhost:11434`），复用现有 `HttpPostFn` 注入缝——对现有异步评估队列、DB、去重逻辑零改动，"本地模型只是另一个 provider"（`docs/M4_Brainstorm.md` 已拍板的设计方向）。
- `get_api_key` 目前强制要求环境变量里有云端 provider 的 key（`core/ai/ai.cpp:26-31`），`Provider::Local` 路径要能在没有配置任何云端 key 的情况下工作（本地不需要 key）。
- CLI 侧 `--provider` 解析（`cli/commands/commands.cpp:316-320`）、`to_string`、`Settings.ai_provider` 默认值逻辑同步扩展，能用 `--provider local` 跑通一次真实评估。
- 硬件约束已确认：M 系列芯片、16GB 内存或以下——优先测轻量级视觉模型（Moondream2 这类 2B 参数以下的，或量化过的 Qwen2.5-VL 小尺寸版本），7B 级别的多模态模型（Qwen2-VL、LLaVA 13B 等）大概率跑不动或体验很差，不作为首选候选；MiniCPM-V/Florence-2 等其它候选（`docs/M4_Brainstorm.md` 已列出）的适配优先级同样让位于轻量模型，具体选型细节留给 Eng Design 阶段结合实测吞吐/质量决定。

**非目标（本周不做）**：
- Tier 0（Apple Vision 确定性信号：曝光/对焦/水平角/裁切建议/embedding）——技术栈完全不同（Obj-C++ 桥接），单独作为后续目标。
- 不解决"照片不出机"这类硬隐私约束的产品判断（`docs/M4_PRD.md` 已有的未决问题），本周只解决"能不能不受云端配额限制地跑评估"这一具体痛点。

**验收标准**：
- `pzt eval --provider local` 在本地跑起来的 Ollama 服务上能跑通一次真实评估（非解码失败的降级路径），产出结构跟云端 provider 一致（`exposure_score`/`composition_score`/`focus_score` 等字段对齐现有 schema）。
- 没有配置任何云端 API key 的环境下，`--provider local` 能正常工作（不因为 `get_api_key` 卡住）。
- 现有 `--provider claude`/`--provider gemini` 行为不受影响（回归测试覆盖）。

## 目标二：Recipe/滤镜系统扩展

**问题**：M1 交付的是验证机制用的 proof-of-concept（`core/recipe/recipe.cpp:48-51` 注释原话），只有 2 个预设（`Origin` 占位 + `Warm`）、`VersionParams` 只有 4 个参数（highlights/shadows/wb_shift_r/wb_shift_b），没有几何变换，没有导入导出格式，没有 LLM 辅助配方生成——离"能真正拿来用"的滤镜系统还有相当距离。

**范围（已拍板：全部四项都要）**：
1. **多加预设 LUT**：沿用现有 4 参数 + LUT 机制（`make_XXX_lut()` 模式），扩充预设库到有实际可选性的数量（具体数字、风格方向留 Eng Design/PRD 细化时定）。
2. **扩展 `VersionParams`**：新增饱和度、对比度、颗粒等参数（具体参数清单留 Eng Design 定，需要评估每个新参数对应的渲染管线改动量）。
3. **补几何变换**：裁切、水平矫正——这是 M3/M4 文档里反复标记但一直搁置的缺口（`docs/M3_PRD.md:34,120`、`docs/M4_Brainstorm.md:45,172,194`、`docs/M4_PRD.md:118`），也是 agent 侧 `Style` Stage 的前置阻塞项。
4. **数值规范化**：统一现有及新增参数的取值口径（比如统一成百分比，或统一成粗粒度档位如 -4 到 4），而不是每个参数各有各的量纲——这既是可用性要求（用户/LLM 更容易理解和生成配方），也是给"目标三"里 LLM 辅助选风格铺路。**已确认不做数据迁移**：本地数据库里没有值得保留的已存用户 version，数值口径切换时旧的 version 数据直接作废重设，不用为迁移单独设计脚本或测试。

**非目标（本周不做）**：
- recipe.json 导入导出格式 + LLM 辅助生成配方的完整工具链——这是"工具能力"而不是"滤镜数量"，跟以上四项是不同维度，本周的选择范围里没有勾选，留后续。
- 风格分享/市场等生态类功能。

**验收标准**：
- 预设数量、可调参数数量相比现状（2 个预设、4 个参数）有实质性增长，具体数字在 Eng Design/细化 PRD 阶段定死。
- 裁切/水平矫正能通过 `set_image_recipe` 同一条路径应用并渲染，不是游离在 recipe 系统之外的另一套机制。
- 所有参数（新旧）取值口径统一，有一份清楚的"参数 → 取值范围 → 语义"对照说明（供后续 LLM 生成配方时当 schema 用）。
- 现有 `Origin`/`Warm` 两个预设、既有测试覆盖的行为不因数值规范化而破坏（已确认不涉及用户历史 version 数据迁移）。

## 目标三：Agent 支持 Apply Recipe（LLM 自动选风格）

**问题**：`docs/M4_Agent_Workflow_Design.md` 的 Stage 库里 `Style` 从一开始就在设计里（"加能力=加一个 Stage 类型"），但从未真正实现——`agent/stages/` 目前只有 Ingest/Evaluate/Dedup/Curate/Deliver 五个。核心侧 `set_image_recipe` API 已经存在且天然幂等，只是没有暴露成 headless 命令。

**范围（已拍板：LLM 根据照片内容自动选一个，不是用户点名）**：
- 新增 `pzt recipe apply <image_path> <recipe_name>` 一类 headless 命令，包成 `core::api::set_image_recipe` 的薄壳（沿用 `pzt eval`/`pzt curate` 已有的 headless 命令面模式）。
- 新增 `agent/stages/style.py`（`Style` Stage），依赖 `Curate` 的产出（选中的图），对每张（或整批）选中图，用 LLM 看图内容 + 目标二产出的预设库，选一个合适的风格并调用上面的 headless 命令应用。
- **复用现有"调整"架构**：`docs/M4_Agent_Workflow_Design.md` 已经设计好"结构化配置增量 + 受影响 Stage 集合"这套调整模型（"换掉第3张"→`Curate`(排除,补1张) 是现成先例）。用户对着 LLM 选的风格说"换个更暖的"这类调整，应该走同一套"作废 `Style`（及下游）、重跑子图"路径，不是一个新发明的交互范式。
- 默认闸门：`docs/M4_Agent_Workflow_Design.md:73` 已经标注 `Style` 因为"主观审美"默认应该带闸门（`courtesy` 或 `required`，留 Eng Design 定档）。

**非目标（本周不做）**：
- 用户在意图消息里直接点名风格名字的手动路径（"用暖色风格"）——已拍板选自动选，手动指定作为可能的后续增强，不在本周设计范围，但 Eng Design 阶段应该确认自动选的实现方式不会让手动指定变得更难加（接口层面留个口子即可，不用现在就做）。
- 风格"强度"级联调整（比如"暖色但淡一点"）——这类精细调整依赖目标二的数值规范化先落地，本周不单独设计。

**验收标准**：
- 端到端：一个 Run 的 `Curate` 产出选中图后，`Style` Stage 能对每张图选出一个预设库里的风格并成功应用（`set_image_recipe` 调用成功、DB 里能查到图片挂上了对应 recipe）。
- LLM 选风格的结果经过闸门（复核）才真正影响最终交付，不会绕过人在环。
- 对 `Style` 的结果不满意时，走"对话调整→重跑 `Style`→不重跑 `Evaluate`/`Dedup`/`Curate`"这条路径，验证方式对齐此前"调整只重跑子图"的测试模式。
- 目标二只做出 2-3 个预设的情况下，`Style` Stage 依然能正常工作（不假设预设库一定很大），只是选择空间小；不因为预设少而报错或卡住。

## 目标四：部署——CLI 一键安装脚本 + 文档网页

**问题**：仓库里没有 README、没有安装脚本、CMake 没有 `install()` 规则、没有 CI、没有文档网站、甚至没有 CHANGELOG.md——一个新用户目前完全没有除了读 `AGENTS.md`（这是给 agent 看的开发指令，不是用户文档）之外的任何上手路径。构建依赖 Homebrew（sqlite3、libraw+libomp 手动 workaround、curl、nlohmann_json），且因为 CoreGraphics/ImageIO 框架天然只支持 macOS。

**范围（已拍板：本周只打包 CLI，agent 部署下周再做，但明确是要做的）**：
- **Homebrew tap 分发（已确认）**：自建一个 Homebrew tap，假设用户已装 Homebrew，用户执行 `brew tap <tap 名称> && brew install pzt` 完成安装（tap 具体仓库/命名留 Eng Design 阶段定）；formula 内部是"描述源码构建步骤（含 libraw+libomp workaround）"还是"分发预编译 bottle"留 Eng Design 决定，但不再考虑脱离 Homebrew 生态的独立签名/公证二进制分发路径。
- **README.md**：项目根目录补一份面向新用户的说明（是什么、能干什么、怎么装、怎么用最基本的流程），跟 `AGENTS.md`（agent 开发指令）明确分工。
- **文档网页**：一个能访问的静态站点（工具选型、内容深度——只到"安装+快速上手"还是覆盖完整功能手册——留 Eng Design 阶段细化）。
- macOS-only 这件事在安装脚本和文档里要显式说明，不能让用户装到一半才发现装不上。
- **CHANGELOG.md（已确认要建，结合本目标一起做）**：Homebrew tap 分发本身就需要版本标签，顺势把 CHANGELOG 建起来；具体生成方式（手写 vs 接 `git-cliff`/`conventional-changelog` 一类基于 commit message 自动生成的工具）留 Eng Design 决定，建立之后遵循项目既有约定——只能通过约定好的生成方式更新，不手动改。

**非目标（本周不做）**：
- agent（Telegram bot）对外分发——需要解决多用户/多 chat_id、每人自己的 API key（云端 provider 或本地 Ollama）配置、进程托管（谁来 7x24 跑这个进程）等本周没有讨论过的问题，明确留到下周单独立项。
- CI/CD（自动化构建发布流水线）——一键安装脚本可以先手动触发验证，CI 是否本周做取决于 Eng Design 阶段评估工作量，先不承诺。

**验收标准**：
- 一台全新的、只装了 Xcode Command Line Tools（或指定的最低前提）的 Mac，跑一键安装脚本后能成功执行 `pzt --help`（或等价的冒烟验证）。
- README 里的安装步骤和脚本实际行为一致（不是脚本一套、文档另一套）。
- 文档网页可访问，内容跟 README 不是简单复制粘贴（网页承载更完整的内容，README 是精简入口）。

## 目标五：Agent 运行时重构——consumer/worker 双线程（2026-07-17 追加）

**问题**：`run_telegram.py` 主循环完全同步，`handle_message` 内嵌整条 drive（Evaluate 本地模型动辄数分钟）和多处同步 LLM 分类调用（本地 Ollama 首响可达 10s+），期间所有入站消息堵在 transport 队列里：连发照片无回应、问询无法应答、"取消"不生效，agent 长时间沉默，只能靠静默提醒兜底。这是 M4 Telegram 闭环真机使用后暴露的最大体验问题（`docs/history/M4_Eng_Design.md` 当时已把"pzt eval 同步批处理的时长/体感"列为已知风险、"更细粒度进度回报"列为遗留开放问题）。本目标的两个诉求：**用户体验**（秒回、可取消、有量化进度）与**可扩展可维护性**（新能力 = 新 Stage + 新 Job/Event，不再往单循环里塞逻辑）。

**范围（已拍板：秒级可杀取消 + 挂本周追加 + 并行 2.0 入口）**：
- 新增 `agent/session/` 模块，双线程运行时：**consumer 线程**（唯一对话线程——消息秒级反应、关键词快路径、timers、应用 worker 事件更新视图并回复用户）+ **worker 线程**（单队列串行执行 LLM 分类/编排 job 与 pzt drive job，闸门处顺带导出并发送预览）+ **SessionView 内存视图**（consumer 私有、事件驱动更新、重启从 RunStore 重建；`RunState`+`RunStore` 仍是唯一持久真相，DriveJob 活跃期间该 run 的变更与落盘归 worker 独占，其余时间归 consumer——所有权协议，杜绝整文件覆盖写互相清掉）。
- **取消语义**：任何状态下"取消"关键词秒回确认（含现在缺失的 Collecting 态）；drive 进行中由 worker 终止当前 pzt 子进程（Evaluate/Dedup 子进程级可杀、StyleApplyAll 循环边界生效；Ingest 等不可中断短步骤跑完为止），`pzt_client.py` 增加可取消调用。eval 逐张落库、重跑自动跳过已评估，杀子进程安全。
- **量化进度**：Evaluate 进行中 consumer 定时轮询 `pzt images --json`（已逐张带 `evaluated` 字段）统计 N/M 并按节奏播报——零 core/cli 改动，headless"原子命令、不做流式输出"契约不动；StyleApplyAll 由 worker 在逐张调用间上报。
- **崩溃续跑增强**：启动时扫描 active run，RUNNING 自动入队续跑并通知用户（现状要等下一条用户消息才续）。
- **并行落地**：新入口 `run_telegram2.py`，旧入口与 `router/session_router.py` 原样保留、随时可切回（两个入口不能同时连同一个 bot，getUpdates 只允许一个消费者）；真机验证通过后转正、删除旧实现。Driver、全部 stages、RunStore、`router/collecting.py`、`compose/*`、`transport/*` 复用不动，run_intent/run_watchfolder 不受影响。

**非目标（本周不做）**：
- 多会话/多 chat_id 并发——属于 agent 对外分发的范畴，随下周部署目标单独立项；单聊天单活跃 run 语义不变（RUNNING 期间新照片进 `_pending` 排队、run 结束并入新批次，对齐现有 AwaitingGate 行为）。
- transport 出站队列/限流、Dedup 量化进度、全 asyncio 化——除非真机验证暴露必要性，否则不做。
- `pzt eval` 的结构化进度输出协议（M4 遗留开放问题）——被 `pzt images` 轮询方案绕开，仍留作以后的可选优化，不在本目标解决。

**验收标准**：
- 连发 20 张照片期间任意时刻发问询，秒级得到回应（不被收图/处理阻塞）。
- Evaluate 进行中发"取消"：立即收到回执，数秒内 pzt 子进程真正终止、run 置 CANCELLED。
- Evaluate 进行中每 60s 至少一条量化进度播报（已评估 N/M 张）。
- 处理进行中新发的照片不丢：进 `_pending`，run 结束并入新批次并提示。
- 进程被 kill -9 后重启：RUNNING 的 run 自动续跑并补发应有的闸门/进度提示，不需要用户先说话。
- 旧测试全绿（并行期间旧实现不动）；新增 session 测试覆盖 job 串行、取消、generation 过期丢弃、事件应用、timers、启动续跑。

## 执行顺序

四个目标已拍板顺序：**目标一 → 目标二 → 目标三 → 目标四**。

- **目标一（本地模型）优先**：直接解除本周最痛的具体阻塞——本地开发/测试目前被云端 API 限流卡住，`Provider::Local` 落地后，后续三个目标开发过程中本身要跑的评估/测试也能受益，不用等到专门腾时间才能测。
- **目标二（recipe 扩展）其次**：为目标三准备好有实际可选性的预设库——目标三"LLM 自动选风格"的价值直接取决于目标二产出的预设数量，顺序上目标二先做正好对应这条依赖。
- **目标三（agent 自动选风格）第三**：依赖目标二的产出，Eng Design 阶段可以提前做（接口、Stage 骨架、闸门设计不依赖预设数量），但端到端验收要放在目标二有实质产出之后。
- **目标四（部署）最后**：四者里最独立，不阻塞、也不被其它三个目标阻塞；本周本身只做一半（CLI 打包，agent 分发留到下周），优先级最低。
- **目标五（agent 运行时重构，2026-07-17 追加）**：依赖目标三（agent Style）已完成的现状，与目标四完全独立、可插队先行——它源于前三个目标交付后真机使用的直接反馈，且纯 Python agent 子系统内改动。对应 Eng Design 单独一份：`docs/W2026-07-15_AgentRuntime_Eng_Design.md`。

**文档拆分建议**：鉴于四个目标分属四个不同子系统（C++ AI provider / C++ 色彩渲染管线 / Python agent / 打包与网站），建议 Eng Design 阶段拆成四份独立文档而不是一份大文档——`docs/W2026-07-15_LocalModel_Eng_Design.md`、`docs/W2026-07-15_Recipe_Eng_Design.md`、`docs/W2026-07-15_AgentStyle_Eng_Design.md`、`docs/W2026-07-15_Deploy_Eng_Design.md`，各自独立评审、独立推进，不互相阻塞，可以按上面的顺序依次开工。本开发目标文档（PRD）本身保持一份，四个目标各自成节，方便一眼看到全貌。

## 待 Eng Design 阶段细化的技术选型
