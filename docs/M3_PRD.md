# PicZTream (PZT) Milestone 3 产品需求文档（AI 辅助功能 · 增量一：选片辅助评分——主动触发）

## 背景与问题陈述

`docs/Roadmap.md` 定义的 Milestone 3 是"可插拔异步能力层"，范围包括自动粗修、审美评分、自动打标签、近似重复检测四个能力，核心约束是不能影响 culling 主流程的零延迟体验，AI 推理必须异步，结果作为建议写回数据库供用户参考而不是替代用户判断。

这份 PRD 最初以"审美评分"（一个笼统的 1-100 分 + 文字点评）立项，主动触发链路（发起请求、异步执行、结果落库、浏览时看到）已经实现并验证可用。在实际使用过程中重新梳理了 AI 在 PZT 工作流里该管什么：PZT 的工作流分两段——**选片+预处理**（挑出曝光正确、构图合理、对焦准确、没有重复的照片，修正明显的技术性错误）和**风格化**（个人审美/风格偏好主导的调色）。风格化高度主观，AI 的判断不见得符合用户自己的审美，不适合让 AI 介入；选片+预处理这一段的判断标准相对客观（曝光/构图/对焦是否达标、有没有重复），AI 更适合在这里帮上忙。

因此这份 PRD 把 AI 的 scope 收窄、重新定义：**从"审美评分"改成"选片辅助评分"**——只做曝光、构图、对焦三个维度的技术性评估，不再评价色彩、情绪表达这类风格化的东西。这个改动沿用同一条已经验证过的主动触发链路，只是评分的内容和输出形状变了，不影响触发方式本身。

## M3 目标（本增量范围）

* 用户可以在 `pzt open` 浏览界面里，对当前正在看的图片主动发起一次 AI 选片辅助评分请求
* AI 从曝光、构图、对焦三个维度分别给出 0-10 的评分和简短原因，评分维度和标准由系统固定的模板决定（评估的是技术层面的问题，不涉及色彩、情绪表达这类风格判断），用户只需要输入可选的"额外指引"来补充/调整这次想让 AI 侧重关注的点
* 系统在收到三个维度的评分后，自行计算：
  * **综合分数** = 三项的平均值，四舍五入取整（不问模型要综合分数，避免模型自己给的综合判断跟三项分数不自洽）
  * **是否达标** = 三项都不低于 6 分（阈值定为 6，任意一项低于阈值即视为不达标）
* 曝光、构图两个维度，AI 额外给出一个修正建议（仅供参考，不自动应用）：
  * 曝光：一个百分比调整量（比如"+15%"表示建议整体调亮 15%）
  * 构图：旋转角度 + 上下左右四个方向的裁切百分比
  * 对焦无法通过后期修正，不提供修正建议
* 以上所有结果异步返回后写入数据库，关联到当前图片，下次浏览重绘时能在信息栏看到
* 调用期间不阻塞浏览、打标签、翻页等其它操作
* 同一张图片的请求还在处理中时，再次发起会被明确告知"处理中"，不会重复调用
* 支持 Claude 和 Gemini 两家云端模型（具体默认使用哪一家、以后怎么在两者之间切换属于实现细节，这份 PRD 不锁定）
* `:` 命令行是一个整体的小命令语言，**所有输入必须以 `/` 开头**（见"触发入口"一节），支持三种形状：
  * `/ai_eval [额外指引]`——不带范围标记，对**当前图片**发起一次评分请求（取代了最初"裸文本直接提交"的做法）
  * `/ai_eval * [额外指引]` / `/ai_eval #标签名 [额外指引]`——对项目里全部图片、或者带指定标签的图片，批量提交评估请求（跳过已经评估过的，只处理还没评估过的），额外指引可选，套用到这一批里的每一张
  * `/tasks`——查看当前有多少评估请求在排队、有没有正在处理，不需要一张一张看进度

## 非目标

* **不做风格判断**——曝光、构图、对焦是这次唯一的评估维度，色彩、光线氛围、情绪表达这类风格化的东西明确排除在 AI scope 之外，交给用户自己在风格化阶段决定
* **不自动应用修正建议**——曝光/构图的修正建议只是展示给用户看的参考数字，不接入渲染管线、不自动改图。真正应用（比如把裁切/旋转套进 `core::recipe` 渲染出来）涉及新增几何变换能力（现在的 recipe 管线只有色调/白平衡参数，没有旋转裁切），是一个复杂度和风险都高得多的独立能力，留到后续单独评估
* **不做本地模型**——效果、离线能力、成本的取舍已经讨论过，见"风险与待确认问题"
* **不做自动介入**——本增量只做主动触发，"评分请求由系统自动发起"是这个能力的下一个增量，触发条件、频率控制留到那时候单独设计
* **不做近似重复检测**——这是一个独立功能，本质是本地图片相似度比对算法（不经过云端 AI），不是这次评分链路的一部分；未来实现时会用这次的综合分数在重复组内挑出保留哪一张，但检测算法本身是完全独立的一次增量
* **不做自动打标签**——留到后续增量单独立项，这份 PRD 不覆盖
* **不做多供应商之间的交互式切换 UI**——这次固定一个默认供应商，不提供"这次用哪家"的选择界面

## 核心用户故事

* 用户浏览到一张图，想知道"这张照片技术上有没有问题、值不值得留"，按 `:` 输入 `/ai_eval` 直接回车，让 AI 用固定的标准分别评估曝光、构图、对焦；也可以在后面加几个字的额外指引（`/ai_eval 重点看看天空有没有过曝`），补充这次想让 AI 侧重关注的点
* 提交之后继续正常浏览、打标签、翻页——不需要停下来等
* AI 处理完之后，三个维度各自的分数和原因、系统算出来的综合分数和是否达标、曝光/构图的修正建议都会出现在信息栏里，不需要用户做任何额外操作去"刷新"或者"查看结果"
* 用户看到某一项分数低（比如构图 3 分，"地平线明显倾斜"），可以据此决定是打上废片标签、还是自己去别的工具或者用 recipe 的 highlights/shadows 之类粗修一下——AI 只提供判断依据，不替用户做决定
* 如果用户手快，对着同一张图片还没等第一次请求处理完就又发起了第二次，会得到"正在处理，请稍后"这样的明确提示，而不是被无声吞掉、也不会因此被重复计费或产生两份互相覆盖的结果
* 用户刚导入一批照片，不想一张一张按 `:` 手动评估，按 `:` 输入 `/ai_eval *` 直接回车，整个项目里还没评估过的图片会依次排队评估，用户继续正常浏览、打标签，不需要等；期间想知道进度，按 `:` 输入 `/tasks` 看一眼还剩多少张在排队
* 用户只想批量评估某一批照片（比如先手动给这批打上一个临时标签），按 `:` 输入 `/ai_eval #临时标签名` 就只处理带这个标签的图片，不影响项目里其它照片
* 用户按了 `:`，还没想好要输入什么就直接按了回车（或者忘了打 `/` 就开始打字），系统提示"命令必须以 / 开头"，不会被无声当成对当前图片提交了一次评估请求——这是主动触发这条链路里唯一一个"手快就可能误触发"的入口，这次刻意堵上

## 功能需求

### 触发入口

`pzt open` 里的 `:` 命令行，按 `:` 之后底部 banner 弹出一个文本输入框，初始显示占位提示（列出可用命令的写法），用户一开始输入就会被输入内容整个替换掉。这个入口现在是一个统一的小命令语言的入口，不只是"AI 评分"专用，比如近似重复检测的 `/dedup` 也走这里（见 `docs/M3_Dedup_PRD.md`）。

**输入必须以 `/` 开头**——这是这一版新加的硬性要求，取代了最初"裸文本直接当额外指引提交"的做法：

* 输入以 `/` 开头——解析成一条命令：
  * `/ai_eval [额外指引]`——不带范围标记，对**当前图片**发起一次评分请求（额外指引可选，直接回车不带任何文字也能提交）
  * `/ai_eval * [额外指引]`——批量评估整个项目里还没评估过的图片
  * `/ai_eval #标签名 [额外指引]`——批量评估带这个标签、还没评估过的图片
  * `/tasks`——查看评估队列的状态（排队数量、有没有正在处理），不需要参数
  * `/dedup * | #标签名`——近似重复检测，见 `docs/M3_Dedup_PRD.md`
  * 输入 `/` 开头但不匹配以上任何一条——提示"未知命令"，不做任何事，也不会被误当成额外指引发给模型
* 输入为空（直接回车）、或者非空但不以 `/` 开头——统一提示"命令必须以 / 开头"，**不提交任何请求**。这跟上一版"直接回车 = 提交一次评分请求"的行为是相反的——上一版这个"手快按了回车就误触发评估"的入口被识别为一个真实的可用性问题，这次改成要求显式打 `/ai_eval` 才会真的对当前图片发起请求
* Esc——静默取消，不提交任何请求，也不显示"必须以 / 开头"这条提示（这是唯一真正意义上的"取消"）

`/ai_eval`（当前图片）和 `/ai_eval *`/`/ai_eval #标签名`（批量）共用一条命令名，靠**有没有范围标记**区分走哪条路径——命令名后面第一个词精确等于 `*`，或者以 `#` 开头，才会被当成范围标记；否则不管这段文字长什么样（哪怕看起来像标签名），整段都当成对当前图片的额外指引。`#` 前缀是刻意加的边界——不然没法区分"用户想批量评估一个叫 `sunset` 的标签"和"用户想给当前图片的额外指引是`sunset 天空要不要提亮`"这两种情况；`/dedup` 的标签范围写法跟 `/ai_eval` 保持一致，同样要求 `#` 前缀，整个控制台只有一套"范围怎么写"的规则，不是两套。

### 评分请求与结果

评分任务由系统固定的模板描述——要求模型从曝光、构图、对焦三个技术维度分别评估这张照片，每个维度给出一个 0-10 的整数分数和一句简短原因；曝光、构图两个维度额外给出一个修正建议（曝光：调整百分比；构图：旋转角度+四个方向裁切百分比），对焦不需要给修正建议。用户输入的"额外指引"（如果有）拼接在这个固定模板之后一起发给模型。

模型返回三个维度的分数+原因（+曝光构图的修正建议）之后，系统自己计算综合分数（三项平均，四舍五入）和是否达标（三项都 ≥ 6）——这两个值不问模型要，保证跟三项分数逻辑自洽。

所有结果写入数据库，跟当前图片关联；浏览时（不需要用户主动刷新）能在信息栏看到。

### 批量评估与任务状态

`/ai_eval * [额外指引]` / `/ai_eval #标签名 [额外指引]`：确定范围（整个项目，或者带指定标签的图片）之后，逐张检查是不是已经评估过——**已经评估过的直接跳过，不重新评估**，哪怕这次想用不同的额外指引也不例外（重复评估目前只能回到单张的 `:` 流程，逐张手动做；批量命令的定位是"把还没处理过的都处理一遍"，不是"用新指引覆盖旧结果"）。还没评估过的依次提交给评估队列，额外指引（如果有）套用到这一批里的每一张。命令提交之后立即返回，不等待这一批全部完成，用户继续正常操作；`/tasks` 用来看进度。

`/tasks`：显示当前评估队列的状态——排队中有几个、有没有正在处理中的（这次不展示具体是哪几张图片，只要数量和状态）。

### 并发与去重

系统需要能区分"这张图片当前有没有一个还没处理完的评分请求"。如果有，用户再次对同一张图片发起请求时，系统直接提示"正在处理，请稍后"，不发起新的调用、不排队等待前一个完成后自动重试。批量命令提交的每一张图片也走这同一条去重逻辑——如果某张图片恰好已经在队列里（比如批量命令跑到一半又手动对其中一张按了 `:`），不会被重复提交。

## 非功能需求

* **延迟：** AI 调用是网络请求，天然是秒级到十秒级的延迟——这个延迟只发生在等待评分结果本身，不能拖慢或阻塞浏览、打标签、翻页这些跟 AI 调用无关的操作
* **可插拔：** 用户完全不碰 `:` 键的话，日常浏览体验要跟没有这个功能时完全一样，不能有任何感知上的差异

## 技术方案概要

本增量的具体 schema、模块划分等技术决策留给 Eng Design 阶段确定，这份 PRD 只锁定产品行为。

## 验收标准

* [ ] 可以在 `pzt open` 里对当前图片按 `:` 输入 `/ai_eval` 发起一次 AI 评分请求，请求期间可以正常导航到其它图片、打标签，不被阻塞
* [ ] 按 `:` 后能看到占位提示（列出可用命令写法），开始输入后提示被输入内容替换掉；`/ai_eval` 后不带额外指引、直接回车，也能正常提交一次评分请求
* [ ] 按 `:` 之后直接回车（什么都不输入），或者输入不以 `/` 开头的文本，都会提示"命令必须以 / 开头"，**不会**提交任何评分请求——验证"手快误触发"的问题已经堵上
* [ ] 同一张图片有请求正在处理时再次发起，得到"处理中"的明确提示，不会重复调用
* [ ] 评分完成后，曝光/构图/对焦三项各自的分数和原因、综合分数、是否达标，都能在信息栏看到，不需要用户手动刷新
* [ ] 曝光、构图的修正建议能看到（数值合理，不是明显乱码/离谱的值）；对焦没有修正建议
* [ ] 综合分数确实是三项的平均值四舍五入，不是模型自己给的另一个数
* [ ] 三项里只要有一项 < 6，"是否达标"能正确显示为不达标
* [ ] Claude 和 Gemini 两种供应商都能真实调通，拿到有效的结果
* [ ] `/` 开头但不匹配任何已知命令的输入，会提示"未知命令"，不会被当成 AI prompt 发给模型，也不会导致程序出错或无响应
* [ ] prompt 确认不再要求模型评价色彩、情绪表达这类风格相关的东西
* [ ] `/ai_eval *` 能正确跳过已经评估过的图片，只对还没评估过的发起请求；提交后立即返回，不阻塞浏览
* [ ] `/ai_eval #标签名` 能正确把范围限定在带这个标签的图片，不影响项目里其它图片；不带 `#` 前缀的裸文本（比如 `/ai_eval 标签名`）会被当成对当前图片的额外指引，不会被误当成范围
* [ ] `/tasks` 能看到当前排队数量和是否有正在处理中的，数字随着评估陆续完成正确减少
* [ ] 批量命令提交的图片如果已经在评估队列里（比如跟单张手动触发的请求撞了），不会被重复提交

## 风险与待确认问题

* **修正建议的可靠性**：旋转角度、裁切百分比这类精确几何量，视觉语言模型不一定擅长给出精确数值（判断"歪不歪"是定性判断，判断"歪几度"是定量判断，两者对模型的要求不一样）——这次先把值存下来、展示出来，真机测试观察这些值靠不靠谱，不夸口保证精度
* **修正建议如何真正应用**：现在的 `core::recipe::VersionParams` 只有 highlights/shadows/wb_shift_r/wb_shift_b 这类色调参数，没有旋转、裁切这类几何变换能力——要把修正建议真的套进渲染管线，需要先给 recipe/渲染管线加一整层新的几何变换支持，工作量和风险都不小，这次明确不做，留到以后单独评估
* **云端 vs 本地的取舍**：跟上一版一样未决——效果好、迭代快 vs 离线不可用、有调用成本、照片数据出本机，这三点跟项目"零网络依赖""本地优先"的倾向有张力，留到自动介入阶段结合真实使用体验再定
* **自动介入的触发条件**：`pzt new`/`pzt rescan` 扫描到新图片时自动排队评分？`pzt open` 空闲时对当前池子里没评过的图自动补跑？要不要总开关或者频率限制？都还没讨论，留到本增量验收之后再单独设计
* **近似重复检测**：已经单独立项，见 `docs/M3_Dedup_PRD.md`
* **自动打标签**：还没讨论，留到后续
* **多供应商切换**：这次只支持"固定一个默认供应商"，交互方式留到后面再定。**TODO**：眼下这个"固定"是字面意义上硬编码在 `cli/commands/browse.cpp`（`handle_ai_prompt_flow` 里直接写死 `Provider::Gemini`）——只是因为开发时手头只有 Gemini 的 key，不是经过设计的选择。后续至少应该做成可配置（比如环境变量指定默认供应商，不需要一步到位做交互式切换 UI），不能让"用哪家"长期停留在源码里改一行这种状态
* **`/ai_eval #标签名` 的解析边界**：这次假设标签名不含空格（按第一个空格切出 `#标签名` 这个 token，后面剩下的整段当额外指引）——如果用户的标签名本身带空格，这个命令会把标签名切错。这次不处理这个边界情况，真遇到了再说
* **额外指引恰好以 `#` 开头的歧义**：`/ai_eval` 用"命令名后第一个 token 是不是 `*`/`#开头`"判断走批量还是当前图片，如果用户对当前图片的额外指引原本就想以 `#` 开头（比如"#1 priority: 检查天空"），会被误判成一次批量请求（标签名解析成"1"，大概率找不到这个标签，报"标签不存在"而不是真的提交当前图片的评估）。这是引入 `#` 前缀这个设计换来的边界情况，这次接受，真遇到了再考虑要不要加转义写法
