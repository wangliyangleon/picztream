# PicZTream (PZT) 周开发目标（2026-07-21）

## 背景与定位

延续里程碑之上的周开发目标节奏（见 `docs/history/W2026-07-15_PRD.md` 背景一节的约定），本周只有一个主题，但足够大，拆成两个阶段目标推进：**把"客观评价单张照片"和"照片之间的比较"这两件事彻底解耦，并把所有涉及比较的选择从"单张打分再横向比"改成"锦标赛式两两比较"。**

### 两个动机

1. **dedup 太慢**：现在近似重复检测走 dHash 解码 + 时间聚类，阻塞式跑一次成本高；AI 一旦介入比较会更慢。需要把"要不要让 AI 介入比较"变成一个可控的、opt-in 的重路径，而不是默认全量。
2. **单张 AI eval 的评分在图片间不可比**：现在 `eval` 产出 exposure/composition/focus 三维 0-10 分，`overall_score`（三维平均）被 dedup 的 keep 选择和 curate 的质量排序当作**唯一的跨图质量信号**。但一张照片单独送进模型打出来的分，本质是在各自独立的会话里生成的，跨图直接比大小并不可靠——构图几乎相同的两张常拿到完全一样的三项分（M3 dedup 设计里已经踩到这个坑，退化成按时间选）。

### 核心设计哲学（本周所有改动的总纲）

- **AI eval = 客观评价当前单张照片**：产出一段简练的文字描述（涵盖构图、色彩、对焦、摄影审美），外加一个"是否存在某方面硬伤而不可用"的布尔 flag（维持现在的 auto-reject）。**不再产任何跨图可比的分数。**
- **涉及比较的选择（dedup 保留哪张、curate 选哪几张）= 锦标赛**：把候选两两送进**同一次 AI 会话**做视觉比较，以单淘汰锦标赛拿到胜者，而不是给每张单独赋一个分数再横向比较那个分数（那个分数并不可比）。

### 已拍板的关键决策（2026-07-21）

1. **非 AI 分簇底层**：本周复用现有 dHash + 时间窗口聚类，不引入新依赖。Apple Vision featureprint 作为"评估其聚类质量相对 dHash 的优劣"单列一条进 `docs/Task_Pool.md`，不在本周范围。
2. **锦标赛分层**：pairwise 视觉比较（发两张图、返回哪张更好）按工程契约"视觉推理归 core"落在 `core/ai`，包成新的 headless 命令。~~bracket 推进（谁跟谁比、如何晋级）落在 agent（Python）~~ **（2026-07-22 修订）整个锦标赛（分簇 + 场次推进 + 判定胜者）都收进 core，一次 headless 调用做完**——bracket 推进是纯确定性算法，不含需要 agent 判断的业务逻辑，收进 core 后 dedup/curate 的 AI 模式共用同一套实现，也避免 agent 侧要发 N 次 `pzt compare` 子进程调用带来的取消粒度问题。agent 只决定"要不要走 AI"和"AI curate 最终从簇 winner 里挑 N 张"。详见 `docs/W2026-07-21_Tournament_Eng_Design.md`。
3. **范围与切分**：全部本周做，分两阶段——**目标一（地基）**先把 eval 解耦、auto-reject 改判据、移除跨图分数下游依赖、建好 core pairwise 能力；**目标二（接线）**再落 dedup 两类、curate 两模式、全局 AI 开关与 agent 编排。
4. **eval 收紧为手动功能（2026-07-21 补充）**：eval 太慢、且对选片的唯一贡献是排废片，而废片假定在锦标赛里也会被淘汰——因此 **agent 不再对整批照片跑 eval**，eval 降级为 pzt CLI 人类用户的手动触发查看（`pzt open` `/ai_eval`、headless `pzt eval`）。选片/dedup 默认排除废片的逻辑**保留**，但只看**废片标签**（来自手动 eval 的 auto-reject 或用户手动打标），不依赖任何 eval 记录。连带：curate/dedup 与 eval 记录彻底解耦（目标一，见该节 dedup/curate 小节的收紧）；agent 删除整批 Evaluate stage、全局开关不门控 eval（目标二）。

## 目标一（地基）：eval 解耦重构 + core pairwise 比较能力

**问题**：现在 `image_evaluations` 表存三维分数 + 各维 note + 曝光/构图修正建议 + 一句 comment；`ai::overall_score()`/`ai::passes_gate()` 是这套分数对下游的唯一出口——`passes_gate`（三维都 ≥6）驱动 auto-reject（`auto_ai_reject` 设置）和 curate 候选过滤（`core/curate/curate.cpp:46`），`overall_score`（三维平均）驱动 dedup keep 选择和 curate 质量排序（`curate.cpp:84,177`）。这套分数正是动机 2 里"不可比"的东西。本阶段把它整个换掉，并顺势建好 pairwise 比较这个 core 新能力（本阶段先建、暂不接入锦标赛，留给目标二）。

**范围**：

1. **eval 输出重构成"文字 + 硬伤 flag"**：`request_evaluation` 的产出从"三维分数结构"改成"一段客观文字描述 + 一个 `unusable`（硬伤不可用）布尔 flag"。文字要求模型覆盖构图、色彩、对焦、摄影审美几个方面（prompt 模板约束，具体措辞留 Eng Design）；flag 表示"是否存在某方面硬伤导致这张根本不可用"（严重失焦/严重欠过曝/等，判据留 Eng Design）。曝光/构图的几何修正建议（rotate/crop 百分比）**一并删除**——它们本来只喂给尚未落地、且已顺延的几何变换，没有下游消费者。
2. **schema 重建，不做迁移**：`image_evaluations` 表按新形状重建（删旧的分数/note/fix 列，新增文字列 + flag 列），沿用 M3 既有先例——库里现有的评估数据都是迭代测试数据，无真实用户数据需保留，直接作废重设，不写迁移逻辑。
3. **auto-reject 改判据**：从"`passes_gate` 为假（三维未达标）"改成"`unusable` flag 为真"。auto-reject 的触发链路（`evaluation_worker` 落库后按 flag 决定是否打"废片"标签）本身不变，只换判据。
4. **移除 `overall_score`/`passes_gate` 及其下游，dedup/curate 降级到非 AI 基线**：删掉这两个函数后，当场断掉的两个消费者同步降级——
   - **dedup keep 选择**：去掉"全员评估过按 `overall_score` 最高留"这一支，统一退化成**留 `captured_at` 最新的那张**（时间也相等再兜底 id 最小，保确定性）。并**新增聚类前排除废片**（现状 dedup 无此排除，否则最新的废片会成为 keep、把好邻居打成重复），只看废片标签。这正是目标二"非 AI dedup"要的行为，提前在这里落定。
   - **curate 候选与排序**：候选过滤**改成纯标签排除（非废片非重复），彻底不看 evaluation 记录**（因 eval 不再整批跑，见已拍板决策 4）——未评估图进候选、废片以外的次品交锦标赛淘汰；每簇代表选择和跨簇排序去掉质量分，只保留多样性/时间信号（现有 `greedy_pick` 的 farthest-point 多样性逻辑保留，质量分维度移除）。这是目标二"非 AI curate"的基线。
5. **core 新增 pairwise 比较能力**：`core/ai` 新增"把两张图放进同一次会话、返回哪张更好（+ 一句简短理由）"的能力，包成一个 headless 命令（命名/输出 schema 留 Eng Design）。**本阶段只把能力和命令建好并单测/真机验证，不接入任何锦标赛**——锦标赛编排是目标二 agent 侧的事。
6. **`pzt open` 信息栏展示改造**：现在信息栏展示三维分数 + 达标标记（`cli/commands/browse.cpp:961` 一带），改成展示新的文字描述 + 可用/硬伤状态。

**非目标（本阶段不做）**：

- 任何锦标赛编排、AI dedup、AI curate、全局开关——全部属于目标二。
- Apple Vision 聚类（见 Task_Pool 单列项）。

**验收标准**：

- `pzt eval`（各 provider）产出新形状（文字 + `unusable` flag），落库读回一致；`--auto-reject` 依据 flag 正确打/不打废片标签。
- `overall_score`/`passes_gate` 从代码库彻底移除，无残留调用点；dedup keep 统一按 `captured_at` 最新，curate 候选按 flag 过滤，两者在移除分数后行为正确、测试覆盖更新。
- pairwise 比较 headless 命令能对两张真实图片跑通、返回确定的胜者与理由；单测覆盖请求拼装/结果解析（复用 `HttpPostFn` 注入缝，不连真网络）。
- `pzt open` 信息栏展示新文字 + 可用性，不再出现三维分数。
- 现有 eval 异步队列/去重/落库骨架、`pzt open` poll-only-on-real-result 机制不回归。

## 目标二（接线）：dedup 两类 + curate 两模式 + 全局 AI 开关 + agent 锦标赛编排

**问题**：目标一把 core 降级到"非 AI 基线"并备好 pairwise 能力后，本阶段在 agent 侧把 AI 锦标赛叠上去，并用一个全局开关统一切换选片流程走不走 AI。

**范围**：

1. **core dedup/curate 暴露簇结构**：因为 bracket 在 agent，`core` 的 dedup/curate 要能**只产出簇结构**（哪些图是一簇）而不替 agent 做 keep/选 N 的决定。接口形状（新 headless 命令，还是给现有 `dedup`/`curate` 命令加一个"只分簇不决策"的模式）留 Eng Design。非 AI 路径下 core 仍可自己完成 keep=留最新 / 多样性选 N（目标一基线），保持零 AI 时的现有一站式行为。
2. **dedup 两类**：
   - **非 AI dedup**（现状 + 目标一基线）：dHash + 时间窗口分簇，每簇留 `captured_at` 最新，其余打"重复"标签。
   - **AI dedup**：在非 AI 分簇的基础上，agent 把每个 dup 簇内的图两两送进 pairwise 比较、以锦标赛选出保留的那张，其余打"重复"标签。
3. **curate 两模式**：
   - **非 AI curate**（目标一基线）：更粗地分簇（调大聚类参数，把"同一场景"聚到一起，而非仅"近乎相同"），每簇一张代表，兼顾多样性选 N。
   - **AI curate**：更粗分簇后，agent 让每簇内部跑锦标赛选出簇内最好的一张（胜者），再从各簇胜者里**随机**选 N 张给用户确认。簇数 < N 时如实返回全部胜者（候选不足，沿用现有 `returned < requested` 语义，不报错）。
4. **全局 AI 开关**：选片流程一个统一开关，决定 dedup/curate 走不走 AI 锦标赛——
   - **开**：dedup 走 AI 锦标赛、curate 走 AI 锦标赛。
   - **关**：dedup 走非 AI（留最新、排废片）、curate 走非 AI（粗簇多样性）。
   - **eval 不受这个开关门控**（已拍板决策 4）：agent 流程里根本不再跑整批 Evaluate，eval 只是 pzt CLI 的手动功能。因此本阶段还要**从 agent 编排里删除 Evaluate stage**（`agent/stages/evaluate.py` 及其在 Plan/Driver 里的位置），废片排除改为只依赖既有的废片标签。
   - 开关落在哪一层（agent Plan 参数由 LLM 从意图判断 / 持久 Settings / 两者结合）留 Eng Design。
5. **agent 接线**：新增 pairwise 比较的 `pzt_client` 调用；在 agent 侧实现锦标赛 bracket 编排（单淘汰、晋级、每对一次 pairwise 调用）；Dedup/Curate Stage 按开关分岔走 core 一站式（非 AI）或"core 分簇 + agent 锦标赛"（AI）；全局开关贯穿 Plan 组装与校验。

**非目标（本阶段不做）**：

- Apple Vision 聚类。
- 锦标赛以外的比较拓扑（瑞士轮、全量两两评分排序等）——本周只做单淘汰锦标赛。
- 把随机选 N 做成可复现（seed 化）——除非 Eng Design 判断用户确认重跑的不确定性是真问题，否则先接受随机。

**验收标准**：

- 全局开关关：一个 Run 端到端不产生任何 AI 评价，dedup 按留最新去重、curate 按粗簇多样性选出 N 张，全程零模型调用。
- 全局开关开：dedup 在 dup 簇内用锦标赛选出保留张；curate 每簇锦标赛选出胜者、再随机选 N 张给用户确认；pairwise 调用次数符合单淘汰锦标赛的量级（k 张簇 k-1 次比较）。
- core dedup/curate 的"只分簇"输出能被 agent 正确消费，非 AI 路径下 core 一站式行为不回归。
- 对 curate/dedup 结果不满意的对话调整仍走"作废受影响子图、重跑、不重跑上游"的既有路径（对齐 agent Style 的调整测试模式）。

## 执行顺序

**目标一（地基）→ 目标二（接线）**，强依赖，不可并行：目标二的 AI 锦标赛依赖目标一建好的 pairwise 能力，目标二的非 AI 基线依赖目标一已把 dedup/curate 降级落定。目标一内部，eval 重构（含 schema、auto-reject、下游移除）和 pairwise 能力两件事相对独立，可并行；`pzt open` 信息栏展示放在 eval 重构之后。

**文档拆分**：本 PRD 一份，两目标成节。Eng Design 进入实现前产出，建议按两阶段拆两份（`W2026-07-21_Eval_Eng_Design.md` 覆盖目标一、`W2026-07-21_Tournament_Eng_Design.md` 覆盖目标二），或按子系统进一步拆，定稿时再定。

## 待 Eng Design 阶段细化

- **eval 文字与 flag 的 prompt 契约**：文字如何约束模型覆盖构图/色彩/对焦/审美四个方面且保持简练；`unusable` flag 的判据（什么程度算"硬伤不可用"）与在 schema_instruction 里的表达（自然语言 vs 结构化输出）。
- **pairwise 比较的输出 schema**：胜者标识 + 一句理由的形状；是否禁止平局（bracket 需要确定晋级，倾向强制模型二选一）；从哪些维度比（构图/色彩/摄影审美，对齐 eval 的口径但只做相对比较）。
- **core dedup/curate "只分簇"的接口形状**：新 headless 命令 vs 现有命令加"分簇模式"；簇结构的输出 schema；非 AI 一站式与"只分簇"两条路径如何共存不重复实现分簇。
- **全局 AI 开关的落点**：agent Plan 参数（LLM 判断）/ 持久 Settings / 结合；与现有 `provider`（local/gemini/claude）和 `auto_reject` 这两个 Plan 字段的关系。
- **curate 粗簇参数**：非 AI/AI curate 的"更粗"具体调哪些聚类参数（时间窗口、hash 阈值），相对 dedup 细簇的取值差异，凭经验起步、真机校准。
- **AI curate 随机选 N 的可复现性**：用户确认环节重跑是否需要稳定结果，先接受随机、Eng Design 评估是否 seed 化。
- **锦标赛的 AI 调用成本/延迟**：本地 Ollama 视觉模型下 pairwise 单次延迟、一个粗簇跑完锦标赛的总耗时，真机基准后决定要不要限制簇内规模或加进度播报（复用目标五 consumer 轮询进度的先例）。
