# 500+ 张图验收测试(increment 6.4.7)

一次性验收脚本,不是常规回归测试,不接入 CMake/ctest——回答 M0_PRD.md 验收标准第 2 条:"浏览五百张图片的全过程主观感受无可感知卡顿,并配合简单的延迟日志加以验证"。

本地没有 500 张真实照片,`generate.cpp` 用 `core/tests/decode_test.cpp` 里已经验证过的 CoreGraphics/ImageIO 现场编码方式合成纯色 JPEG,交替用真机测试实测过的手机拍照分辨率(3072x4080/4080x3072),覆盖 `fit_within` 两种朝向。

**已知局限**:纯色 JPEG 压缩率远高于真实照片,解码速度接近最优情况——这个数据集验证的是"500 张规模下没有病态的性能劣化",不是精确的单张解码耗时。真实解码耗时以真机测试实测过的数字为准(RelWithDebInfo 构建下 50-70ms/张,Debug+ASan 构建下会被放大到 700-800ms,不能用来判断真实体验)。

## 用法

```
# 1. 单独起一个不带 sanitizer 的构建目录(不改项目默认的 Debug+ASan 设置)
cmake -B ../../build_reltest -DCMAKE_BUILD_TYPE=RelWithDebInfo -S ../..
cmake --build ../../build_reltest

# 2. 生成合成数据集(输出到仓库外的临时目录)
clang++ -std=c++20 -O2 -o generate generate.cpp \
  -framework CoreGraphics -framework ImageIO -framework CoreFoundation
./generate ~/pzt_bench_500 550

# 3. 建项目
../../build_reltest/cli/pzt new Bench500 ~/pzt_bench_500

# 4. 跑验收脚本
python3 run_acceptance.py ../../build_reltest/cli/pzt Bench500
```

## 按键序列

一长串 `l` 走完约 500 张(骨架);每 50 张左右插几次 `h` 往回走,压一下预取窗口反向驱逐/重新填充路径;每 100 张左右按一次 `x`(最简单的单键打标签路径,不需要在无头环境里驱动 `space` 那种两步菜单);中途(第 250 张附近)按 `g` `0` 切进筛选视图、再 `g` `g` 清除筛选(这是唯一确认过有固有冷解码成本的操作,顺带能看到 `filter_by_tag` 那行查询耗时日志);最后 `q` 触发退出汇总。

## 判定

脚本解析退出时打印的 `key-to-render summary: n=... avg=...ms p95=...ms max=...ms` 那一行(6.4.7 新加的功能),对照参考阈值:平均 &lt; 30ms(大多数帧应该是预取命中)、p95 &lt; 100ms(容许来回走动和筛选切换)、max &lt; 250ms。这些是给无头验证用的参考阈值,不是 PRD 本身的硬性数字——PRD 原文是主观的"无可感知卡顿"。

## 人工/agent 分工

能完全无头验证的部分:500+ 张走完不崩溃不卡死、汇总数字在参考阈值内、没有随会话推进变慢的趋势。仍然需要在真实 Ghostty 窗口里亲自确认的部分:Kitty 协议实际换图是否真的"感觉"流畅(cli 侧时间戳测不出来 Ghostty 自己处理图片协议命令这一步的观感),以及合成数据集的高解码速度可能掩盖真实照片才有的某些慢点——最终 PRD 验收还是需要用真实照片过一遍主观感受(不需要凑够 500 张真实照片,复用已有的小项目就够,合成数据集是规模层面的补充验证,不是替代)。

## 结果

跑在 550 张合成图(3072x4080/4080x3072 交替)、RelWithDebInfo 构建、按键序列见上(约 520 步 `l`/`h`,`x` 每 100 步一次,第 250 步附近 `g0`/`gg` 筛选切换一次)。

```
n=521 avg=34.32ms p95=36.72ms max=536.34ms
PASS(参考阈值内,仍需真机主观确认)
```

去重后的每帧数据额外确认了两点(用带 `--debug` 的会话手动抓 `key-to-render` 逐帧数值验证,不是脚本本身自动做的):

- **max 值只出现一次,就在会话的第一帧**(打开项目、还没做任何导航就先看到的第一张图)——预取缓存启动时是空的,第一张图不可能被预先解码过,这是任何交互式看图工具都躲不掉的一次性冷启动成本,不是反复出现的问题。除了这一帧,其余所有帧紧密聚集在 35-37ms 附近,没有出现散落的高峰(包括第 250 步那次 `g0`/`gg` 筛选切换,也没有比其它帧显著更慢——推测是因为筛选切换前后 `l` 键已经把预取窗口内的图基本填满,冷切换的代价被摊掉了;真实使用中标签分布更稀疏,筛选切换的冷启动代价可能更明显,这个数据集验证不出那种情形)。
- **没有随会话推进变慢的趋势**:排除首帧后,前 50 帧均值 31.4ms,后 50 帧均值 33.7ms,差距在噪声范围内,没有观察到预取窗口/后台线程随时间推移逐渐跟不上的迹象。

**已知局限,再强调一次**:合成纯色 JPEG 解码速度接近最优情况,这组数字证明的是"550 张规模下、预取机制本身没有病态劣化",不是"真实照片浏览体验一定是这个数字"——真实解码耗时以本项目真机测试实测过的数字为准(RelWithDebInfo 下单张 50-70ms)。最终 PRD 验收签字仍然需要你在真实终端里用真实照片过一遍主观感受。
